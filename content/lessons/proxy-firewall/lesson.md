# Proxy Firewall: ניתוק החיבור הישיר

## למה יש רכיב באמצע בין שכבה 3 לשכבה 7?

בשיעור הקודם ראינו חומת אש קלאסית ([[Firewall]]) שהמרצה ממקם במפורש בשכבה 3 —
היא בודקת את **כותרת** המנה (כתובת מקור, יעד, פורט, פרוטוקול) ומחליטה forward או
drop. בשיעור על ה-[[WAF]] ראינו רכיב שבוחן את **תוכן** החבילה בשכבה 7 ותופס
התקפות web כמו SQL Injection. בין שני אלה נפער פער: איך בכלל אפשר לבחון תוכן, אם
המנות פשוט זורמות דרך חומת האש מהלקוח אל השרת?

התשובה היא רכיב שהמרצה מזכיר בשקופית אחת בלבד ואינו מרחיב עליו — ה-**Application
Gateway (Proxy FW)**. הרעיון שלו פשוט וחזק: אל תעביר את התעבורה — **עצור אותה**.
Proxy Firewall אינו מעביר מנות בין הצדדים; הוא מנתק את חיבור הלקוח, פותח חיבור שני
משלו אל השרת, וכך הופך לנקודה שרואה את **כל** התוכן העובר. זהו המבנה שמאפשר את כל
בדיקת התוכן בשכבה 7 — ולכן הוא הגשר המושגי בין חומת אש בשכבה 3 לבין ה-WAF.


:::objectives
- להסביר את המנגנון המרכזי של Proxy Firewall — ניתוק החיבור ופתיחת חיבור שני.
- להבחין מבנית בין Proxy Firewall שמנתק חיבור לבין [[Firewall]] שמעביר מנות.
- להבדיל בין Forward Proxy ל-Reverse Proxy ולזהות היכן כל אחד ממוקם.
- למקם את ה-Proxy במודל השכבות ולהבחין בינו לבין [[WAF]] ו-[[NGFW]].
- לנמק את המחיר של הגישה: השהיה, סיום TLS, ומודעות לפרוטוקול.
:::

## המנגנון: הפרוקסי מנתק ומחבר מחדש

הדבר היחיד שצריך להבין כדי להבין את כל השאר: **Proxy Firewall אינו מעביר את התעבורה
— הוא מסיים אותה ופותח חיבור חדש בשם הלקוח**. כשלקוח פונה לשרת דרך [[Proxy Firewall]],
נוצרים שני חיבורים נפרדים לחלוטין: אחד בין הלקוח ל-Proxy, ואחד בין ה-Proxy לשרת.
אין שום מנה שעוברת מהלקוח אל השרת כמות שהיא.

> המבנה הזה — שני חיבורי TCP נפרדים עם מתווך שמפרק ומרכיב מחדש

ההבדל מ-[[Firewall]] קלאסי הוא **מבני, לא רק "חזק יותר"**. חומת אש בשכבה 3 מסתכלת
על כותרת המנה ומחליטה להעביר או להשליך — אבל המנה שהיא מעבירה היא אותה מנה שהגיעה,
עם אותו מטען. Proxy אף פעם לא מעביר את המנה המקורית; הוא קורא את הבקשה עד הסוף,
מבין אותה, ואז **כותב בקשה חדשה משלו**. מכאן נובעת כל היכולת שלו: מי שקרא והרכיב
מחדש את הבקשה — ראה בהכרח את מלוא תוכנה.

מכיוון שה-Proxy מדבר את שפת הפרוטוקול (HTTP, FTP, SMTP), הוא יכול גם לאכוף **התאמה
לפרוטוקול** — לחסום בקשה מעוותת או חריגה מהתקן, מה שחומת אש מבוססת-כותרות כלל לא
מבחינה בו. זו בדיוק הנקודה שהמרצה כן רושם על השקופית: "Ensure protocol conformance,
block specific sites".

:::selfcheck
question: מדוע ההבדל בין Proxy Firewall ל-Firewall קלאסי מתואר כ"מבני" ולא כ"רמת אבטחה גבוהה יותר"?
answer: כי הם עושים דבר שונה מהיסוד. חומת אש קלאסית מעבירה את אותה מנה הלאה לאחר בדיקת הכותרת; Proxy אינו מעביר את המנה כלל — הוא מסיים את החיבור, קורא את הבקשה, ומחבר חיבור שני חדש בשם הלקוח. בגלל המבנה הזה — שני חיבורים נפרדים במקום אחד — נולדת היכולת לראות את מלוא התוכן.
:::

## מה הפרוקסי רואה, ואיזה כיוון הוא פונה

הרווח מהניתוק הוא **ראות מלאה לתוכן האפליקטיבי**. חומת אש בשכבה 3 יודעת שמנה הגיעה
מכתובת מסוימת אל פורט 443 — ותו לא. Proxy שמסיים את החיבור רואה את בקשת ה-HTTP
השלמה: את הכתובת המבוקשת, את הכותרות, את הפרמטרים ואת גוף הבקשה. הוא יכול להחליט
לפי כל אלה, לא רק לפי המעטפה.


יש שני כיוונים שבהם מציבים Proxy:

- **Forward Proxy** — יושב מול **הלקוחות** ומתווך את פנייתם החוצה. הוא שולט ביציאה
  לאינטרנט: מסנן אתרים, אוכף מדיניות גלישה, ומסתיר את כתובות הלקוחות הפנימיים. השרת
  החיצוני רואה רק את ה-Proxy.
- **Reverse Proxy** — יושב מול **השרת** ומתווך את הפניות הנכנסות אליו מבחוץ. הוא
  מסתיר את השרת האמיתי, וממוקם לרוב ב-[[DMZ]]: התוקף מדבר עם המתווך, לא נוגע בשרת
  הפנימי. הרעיון של ה-Reverse Proxy הוא הבסיס שעליו נבנה מאוחר יותר ה-[[WAF]].
![[מדריך_פעולת_שרת_פרוקסי_וסוגיו.png]]
הכיוון קובע את התפקיד, אבל המנגנון זהה בשניהם — ניתוק החיבור ופתיחת חיבור שני.

:::selfcheck
question: חברה רוצה גם למנוע מעובדיה לגלוש לאתרים אסורים, וגם להסתיר את שרת ה-web הציבורי שלה מפני תוקפים. אילו כיווני Proxy משרתים כל מטרה?
answer: Forward Proxy מול הלקוחות אוכף את מדיניות הגלישה ומונע יציאה לאתרים אסורים. Reverse Proxy מול השרת, לרוב ב-DMZ, מסתיר את שרת ה-web האמיתי — התוקף מגיע אל המתווך ולא אל השרת. שתי המטרות, אותו מנגנון של ניתוק חיבור, שני כיוונים.
:::

## מיקום במודל השכבות: פרוקסי, WAF ו-NGFW

הוא מציג טבלת השוואה בין **Layer 3 firewalls** ל-**Layer 7 firewalls**: שכבה 3 מסננת לפי כתובות IP ופורטים, מהירה ובעלת קצב גבוה; שכבה 7 "בוחנת את התוכן שבתוך המנות", איטית יותר בגלל בדיקה עמוקה, וממתנת איומים שמנצלים חולשות אפליקציה. ה-Proxy Firewall שייך במובהק לצד ה**שכבה 7**
של הטבלה הזו — הוא הרכיב שמנתק את החיבור כדי לבחון תוכן.

> קישור ה-Proxy אל צד שכבה 7 של הטבלה הוא ההרחבה שלנו על בסיס מנגנון הניתוק; המרצה
> מציג בטבלה "Layer 7 firewall" כללי בלי לפרט שהמנגנון שמאחוריו הוא Proxy.

בתוך שכבה 7 חשוב להבחין בין שלושה רכיבים קרובים:

| רכיב | מה הוא | הטווח שלו |
| --- | --- | --- |
| [[Proxy Firewall]] | השער הכללי בשכבת האפליקציה שמנתק חיבור | כל פרוטוקול שנכתב לו מנוע (HTTP, FTP, SMTP) |
| [[WAF]] | צאצא מתמחה של הרעיון | תעבורת web בלבד, וחתימות התקפה כמו SQL Injection |
| [[NGFW]] | קופסה שמאחדת שכבה 3 ושכבה 7 | חבילה שלמה: סינון מנות, בדיקת תוכן, IPS |

הרעיון המרכזי: ה-Proxy הוא הקטגוריה הרחבה, וה-WAF הוא ההתמחות שלו ל-web. שניהם
בוחנים תוכן בשכבה 7; ההבדל הוא בהיקף. חומת אש בשכבה 3 עדיין נחוצה לצידם, כי היא
עוצרת בזול נפחי תעבורה גדולים ששכבה 7 יקרה מכדי לבדוק.

:::selfcheck
question: על סמך טבלת המרצה בשקופית 30, לאיזה צד — שכבה 3 או שכבה 7 — שייך ה-Proxy Firewall, ומה מייחד אותו מ-WAF באותו צד?
answer: ה-Proxy שייך לצד שכבה 7, כי הוא מנתק את החיבור כדי לבחון את תוכן המנה ולא רק את כותרתה. מ-WAF הוא נבדל בהיקף: ה-Proxy הוא שער כללי לכל פרוטוקול שנכתב לו מנוע, בעוד שה-WAF הוא ההתמחות של אותו רעיון לתעבורת web ולחתימות התקפה אפליקטיביות.
:::

## המחיר: השהיה, סיום TLS, ומודעות לפרוטוקול

לניתוק החיבור ולבדיקת התוכן יש מחיר, וחשוב להכיר אותו כדי להבין למה לא מציבים Proxy
בכל מקום.

> שלושת המחירים שלהלן מתועדים במקורות חיצוניים (NIST SP 800-41 Rev. 1; CompTIA
> Security+ SY0-701; תיעוד יצרנים). המרצה מזכיר רק את עקרון ה"throughput נמוך יותר
> בגלל בדיקה עמוקה" בטבלת שכבה 7 (שקופית 30).

- **השהיה וקצב (latency & throughput)** — פירוק כל בקשה עד שכבה 7 והרכבתה מחדש עולים
  זמן ומשאבים. ה-Proxy איטי מחומת אש בשכבה 3, ולכן אינו מתאים לצווארי בקבוק שבהם
  נפח ומהירות הם קריטיים. זו בדיוק ההבחנה שהמרצה עושה בטבלה: שכבה 3 = throughput גבוה,
  שכבה 7 = נמוך יותר.
- **סיום TLS (TLS termination)** — כדי לראות תוכן מוצפן, ה-Proxy **חייב לפענח** את
  ה-TLS, לבדוק, ולהצפין מחדש אל השרת. בלי זה התוכן נראה לו כרעש. הדבר מחייב אותו
  להחזיק מפתחות ולהיות נקודת אמון — ורכיב אחד שרואה הכל בטקסט גלוי הוא גם יעד יקר ערך.
- **מודעות לפרוטוקול (per-protocol awareness)** — ה-Proxy זקוק למנוע ייעודי לכל
  פרוטוקול שהוא מבין. פרוטוקול חדש או לא-סטנדרטי שלא נכתב לו מנוע פשוט לא ייבדק לעומק.

לכן, בפועל, Proxy Firewall אינו מחליף את חומת האש בשכבה 3 — הוא מוצב במקומות שבהם
בדיקת התוכן שווה את המחיר: ב-Reverse Proxy מול שרת רגיש ב-[[DMZ]], או ב-Forward
Proxy שאוכף מדיניות יציאה. ברוב הפריסות המודרניות היכולת הזו נבלעת בתוך [[NGFW]]
שמאחד את שתי השכבות.

:::selfcheck
question: מדוע העובדה ש-Proxy Firewall "חייב לסיים TLS" היא גם יכולת וגם סיכון?
answer: יכולת — כי בלי לפענח את ה-TLS ה-Proxy כלל לא יכול לראות את התוכן המוצפן, ובדיקת התוכן היא כל תכליתו. סיכון — כי כדי לפענח הוא חייב להחזיק מפתחות ולהחזיק את התעבורה בטקסט גלוי ברגע הבדיקה, ולכן הוא הופך לנקודת אמון מרכזית וליעד יקר ערך שאם ייפול — נחשף הכל.
:::

:::keypoints
- Proxy Firewall אינו מעביר מנות — הוא מסיים את חיבור הלקוח ופותח חיבור שני אל השרת בשמו.
- הלקוח והשרת לעולם אינם מדברים ישירות; זהו ההבדל המבני מ-Firewall בשכבה 3.
- ניתוק החיבור הוא מה שמאפשר לראות ולבחון את מלוא התוכן האפליקטיבי בשכבה 7.
- Forward Proxy יושב מול הלקוחות ושולט ביציאה; Reverse Proxy יושב מול השרת ומסתירו, לרוב ב-DMZ.
- ה-Proxy הוא השער הכללי בשכבה 7; ה-WAF הוא ההתמחות שלו ל-web; ה-NGFW מאחד שכבה 3 ושכבה 7.
- המחיר: השהיה וקצב נמוך, חובה לסיים TLS כדי לראות תוכן מוצפן, ומודעות נפרדת לכל פרוטוקול.
:::
