# Web Shell — הדלת האחורית שיושבת על השרת שלך

## למה הנושא הזה קיים

לאורך הקורס בנינו פרימטר: הצבנו את השרתים הציבוריים ב-[[DMZ]], חצצנו אותם מהרשת
הפנימית בחומות אש, והגדרנו בקפדנות מה מותר להיכנס. אבל יש שאלה אחת שכל הארכיטקטורה
הזו לא עונה עליה: **מה קורה אחרי ששרת ה-web הציבורי — זה שבכוונה חשפנו לאינטרנט —
נפרץ?**

[[Web Shell]] הוא התשובה הקונקרטית לשאלה הזו. זהו סקריפט זדוני קטן שהתוקף משתיל על
שרת ה-web, ומאותו רגע הוא מחזיק דלת אחורית קבועה אל תוך הארגון — דלת שיושבת בדיוק
במקום שבנינו כדי להיות חשוף. כאן כללי התעבורה של יחידת ה-DMZ מפסיקים להיות מופשטים:
פתאום ברור למה חייבים להגביל את מה ש**יוצא** מה-DMZ, ולא רק את מה שנכנס.

הנושא הזה קיבל במצגת המרצה עשרה שקפים — יותר מווירוסים, תולעים וכופרה יחד. הוא
נבחן ישירות ב-2024, והוא ממחיש רעיון שחוזר בכל הקורס: החומה החזקה ביותר שווה מעט
אם התוקף כבר מריץ פקודות מהצד הפנימי שלה.

:::objectives
- להסביר מהו Web Shell ומדוע שלושת מאפייניו הופכים אותו לאיום ייחודי.
- למנות את דרכי ההשתלה של Web Shell ולזהות אותן בתרחיש נתון.
- לתאר מה תוקף עושה לאחר שהשיג שליטה, ולקשר זאת לתנועה רוחבית ברשת.
- להסביר מדוע Web Shell המוסווה כקובץ לגיטימי הוא מהקשים ביותר לזיהוי.
- לבחור אמצעי הגנה מתאימים, ולהבין מדוע הגנת פרימטר לבדה אינה מספיקה.
:::

## מה זה Web Shell, ולמה הוא שונה מכל נוזקה שראינו

עד כה למדנו נוזקות שפועלות על תחנת הקצה: וירוס שנצמד לקובץ, תולעת שמתפשטת ברשת,
דרופר שמוריד payload. Web Shell שונה מהותית — הוא לא רץ על המחשב של הקורבן, אלא על
**שרת ה-web של הארגון**, והוא נכתב בשפה שהשרת ממילא יודע להריץ.

המרצה מגדיר אותו דרך שלושה מאפיינים, וכל אחד מהם תורם לסכנה:

- **מבוסס סקריפט (Script-Based)** — נכתב בשפות צד-שרת כמו PHP, ASP, JSP או Python.
  אין כאן קובץ הפעלה מהודר שאפשר לזהות בחתימה; זהו קובץ טקסט שהשרת מפרש, בדיוק כמו
  כל קובץ לגיטימי אחר באתר.
- **מוסתר לעין כול (Hidden in Plain Sight)** — הוא מונח בין קובצי האתר הלגיטימיים.
  שרת web אמור להכיל קובצי סקריפט, ולכן קובץ אחד נוסף לא מעורר חשד. משתמש רגיל פונה
  לסקריפטים התקינים; התוקף פונה בדיוק לאותה ספרייה, רק לקובץ אחר.
- **שליטה מרחוק (Remote Control)** — הוא נותן לתוקף הרצת פקודות, ניהול קבצים והוצאת
  מידע, הכול דרך בקשות HTTP/HTTPS רגילות.

ההבדל הזה הוא שהופך את ה-Web Shell למסוכן: הוא **נקודת אחיזה קבועה**. וירוס צריך
שמשתמש יריץ אותו שוב; Web Shell פשוט ממתין. התוקף חוזר מתי שירצה, גולש לכתובת של
הקובץ, ומקבל שליטה — בלי לפרוץ מחדש.

:::important
Web Shell אינו "עוד סוג נוזקה". הוא backdoor: לא כלי שגורם נזק חד-פעמי, אלא ערוץ
גישה מתמשך שדרכו כל שאר הנזק מתבצע. הנזק האמיתי מגיע ממה שהתוקף עושה **אחרי**
ההשתלה, לא מהקובץ עצמו.
:::

:::selfcheck
question: במה שונה Web Shell מווירוס מבחינת המקום שבו הוא פועל ומבחינת הצורך באינטראקציה חוזרת?
answer: וירוס רץ על תחנת הקצה של המשתמש ודורש שהמשתמש יפעיל אותו כדי לפעול שוב. Web Shell רץ על שרת ה-web של הארגון, ואינו דורש שום אינטראקציה חוזרת — הוא ממתין באופן פסיבי, והתוקף מפעיל אותו מרחוק דרך בקשת web מתי שירצה. זו נקודת אחיזה קבועה, לא אירוע חד-פעמי.
:::

## איך הוא מגיע לשרת

Web Shell לא מופיע יש מאין — התוקף חייב תחילה להעלות את הקובץ אל השרת. המרצה מונה
ארבע דרכים עיקריות, ושתי הראשונות הן החשובות ביותר:

- **העלאת קבצים ללא הגבלה (Unrestricted File Upload)** — ניצול טפסי העלאה שאינם
  מאמתים את סוג הקובץ. אתר שמאפשר להעלות תמונת פרופיל אך אינו בודק שהקובץ באמת
  תמונה, יקבל בשמחה גם קובץ ‎.php‎ או ‎.asp‎ — וברגע שהקובץ יושב על השרת, גלישה
  אליו מריצה אותו. זוהי דרך ההשתלה הנפוצה ביותר.
- **ניצול חולשה (Vulnerability Exploitation)** — ניצול חולשות מוכרות כדי לכתוב את
  ה-shell לשרת: **RFI/LFI** (הכללת קבצים מרוחקים או מקומיים), **SQL injection**,
  או חולשות ידועות ב-CMS ובתוספים (plugins).
- **גניבת פרטי גישה (Compromised Credentials)** — שימוש בפרטי FTP, SSH או פאנל
  ניהול גנובים כדי להניח את קובץ ה-shell ישירות, בלי לנצל שום חולשה טכנית.
- **מתקפת שרשרת אספקה (Supply Chain Attack)** — הטמעת shell בתוך תוסף, ערכת עיצוב
  (theme) או עדכון תוכנה של צד שלישי שהארגון מתקין מרצונו.

שימו לב לתבנית: כל ארבע הדרכים חולקות מכנה משותף עם שרשראות התקיפה שראינו קודם —
הן מתחילות מפרצה שכבר קיימת בשרת. Web Shell אינו פורץ את השרת; הוא **מנצל** פרצה
כדי להתמקם בו.

:::example
בתרחיש טיפוסי, אתר מסחר מפעיל טופס "העלאת מסמכים" למחלקת התמיכה. המפתחים בדקו את
הסיומת בצד הלקוח בלבד. התוקף עוקף את הבדיקה, מעלה קובץ ‎shell.php‎, וגולש אל
‎https://shop.example.com/uploads/shell.php‎. השרת מריץ את הקובץ — והתוקף מקבל
שורת פקודה על שרת המסחר.
:::

:::selfcheck
question: מדוע העלאת קבצים ללא אימות סוג הקובץ היא וקטור כה מסוכן להשתלת Web Shell?
answer: מפני שהיא מאפשרת לתוקף להניח קובץ סקריפט הרצה (‎.php‎/‎.asp‎) ישירות בתוך תיקייה שהשרת מסוגל להריץ. ברגע שהקובץ שם, עצם הגלישה אליו מפעילה אותו — התוקף לא נדרש לשום חולשה נוספת. לכן ההגנה מחייבת לאמת את סוג הקובץ בצד השרת, לאחסן העלאות מחוץ ל-web root ולשנות את שמות הקבצים.
:::

## מה התוקף עושה אחרי שהשתלט

ברגע שה-Web Shell מותקן, שרשרת התקיפה שהמרצה מציג מונה חמישה שלבים:

1. **איתור (Recon)** — סריקת השרת לאיתור חולשות.
2. **העלאת ה-shell (Upload Shell)** — ניצול פרצת ההעלאה.
3. **ביסוס גישה (Establish Access)** — הרצת ה-shell דרך HTTP.
4. **תפעול (Operate)** — הרצת פקודות מרחוק.
5. **הוצאה והתבססות (Exfiltrate)** — גניבת מידע והשארת אחיזה קבועה.

מרגע שהתוקף בשלב התפעול, ה-shell נותן לו סל יכולות רחב: הרצת פקודות מרחוק,
**קצירת פרטי גישה (Credential Harvesting)** מקבצי תצורה ומהזיכרון, **הוצאת מידע
(Data Exfiltration)**, **השחתת האתר (Website Defacement)** והפניית משתמשים, ובניית
**מנגנוני התמדה (Persistence Mechanisms)** — חשבונות backdoor, משימות מתוזמנות
ו-shells נוספים, כדי שגם אם ה-shell המקורי יימחק, תישאר דרך חזרה.

אבל היכולת המסוכנת ביותר היא הקישור הבא: מהשרת ב-DMZ, התוקף מבצע [[Lateral Movement]]
— הוא סורק את הרשת הפנימית ותוקף מארחים נוספים. וכאן מתחברת יחידת ה-[[DMZ]]: הסיבה
שאנחנו מגבילים את התעבורה ה**יוצאת** מה-DMZ אל הרשת הפנימית היא בדיוק התרחיש הזה.
אם שרת ה-web ב-DMZ נפרץ אבל אינו יכול לפתוח חיבורים אל השרתים הפנימיים, ה-Web Shell
נותר כלוא בשרת בודד. אם התעבורה היוצאת פתוחה — הפריצה של שרת אחד הופכת לפריצה של
כל הארגון.

:::warning
הנטייה היא לראות ב-Web Shell "בעיה של שרת האתר". זו טעות: שרת ה-web הוא רק **נקודת
הכניסה**. המטרה האמיתית של רוב התוקפים היא הרשת הפנימית, וה-Web Shell ב-DMZ הוא
הקרש הראשון בגשר אליה. לכן הכלה של הפגיעה תלויה בסגמנטציה ובהגבלת תעבורה יוצאת, לא
רק בניקוי השרת הנגוע.
:::

:::selfcheck
question: שרת web ב-DMZ נדבק ב-Web Shell. מדוע הגבלת התעבורה היוצאת מה-DMZ אל הרשת הפנימית היא בקרה קריטית דווקא במצב הזה?
answer: מפני שהיכולת המסוכנת ביותר של Web Shell היא תנועה רוחבית — שימוש בשרת שנפרץ כנקודת זינוק לתקיפת השרתים הפנימיים. אם ה-DMZ מוגדר כך שהשרת הציבורי אינו יכול לפתוח חיבורים אל הרשת הפנימית, הפריצה נשארת כלואה בשרת אחד. בלי ההגבלה הזו, פריצת שרת בודד הופכת לדריסת רגל בכל הארגון.
:::

## למה הוא כל כך קשה לזיהוי

זו הנקודה שנבחנה ישירות ב-2024: מבין כל הנוזקות, **Web Shell המוסווה כקובץ לוג
לגיטימי הוא הקשה ביותר לזיהוי** — קשה יותר מווירוס, מתולעת או מכופרה. חשוב להבין
מדוע, כי הסיבה אינה תחכום הקוד אלא **המיקום והערוץ**.

שלושה גורמים מצטברים:

1. **הוא יושב במקום שבו קבצים אמורים להיכתב.** שרת web כותב קבצים כל הזמן — לוגים,
   העלאות, קבצים זמניים. קובץ נוסף בספרייה כזו אינו חריג. אם הוא מחופש לקובץ לוג,
   הוא מתמזג בתוך רעש שממילא קיים.
2. **הוא נגיש דרך HTTPS על port 443.** זהו בדיוק הפורט שכל שרת web חייב לקבל אליו
   תעבורה. אף חומת אש היקפית לא תחסום בקשת HTTPS נכנסת לשרת שכל תכליתו לשרת HTTPS.
3. **התעבורה שלו אינה ניתנת להבחנה מתעבורת web רגילה.** הפקודות של התוקף עוברות
   כבקשות HTTP/HTTPS, לעיתים מוצפנות. מנקודת מבט הרשת, בקשה אל ה-shell נראית כמו
   כל גלישה אחרת.

התוצאה: מנקודת המבט של הרשת, **שום דבר אינו נראה חריג**. זו הסיבה שזיהוי מבוסס
חתימות מתקשה (הקוד יכול להשתנות בקלות, והוא מתחזה לקובץ תקין), וגם זיהוי מבוסס
התנהגות מתקשה (הפעילות עוברת בערוץ ובמיקום שבהם התנהגות כזו היא לגיטימית).

מה כן מסגיר אותו? המרצה מונה סימני פשרה (**Indicators of Compromise, IOC**) שמתמקדים
בשרת עצמו, לא ברשת: קבצים חדשים ובלתי צפויים בספריות ה-web; קבצים עם חותמת זמן
עדכנית חשודה; בקשות **HTTP POST חריגות אל קבצים שנראים סטטיים**; חיבורים יוצאים
בלתי רגילים; ובעיקר — תהליך של שרת ה-web שמוליד ‎cmd.exe‎ או ‎/bin/sh‎, כלומר שרת
שפתאום מריץ פקודות מערכת.

:::important
המסקנה למבחן ולמעשה זהה: את ה-Web Shell **לא תופסים בפרימטר**. חומת האש רואה תעבורת
HTTPS תקינה אל שרת שאמור לקבל HTTPS. תופסים אותו רק דרך ניטור של השרת עצמו — שינויים
בקבצים, תהליכים חריגים, ודפוסי בקשות שאינם מתאימים לקובץ שאמור להיות סטטי.
:::

:::selfcheck
question: מדוע חומת אש היקפית ו-NIDS מבוסס-רשת מתקשים לזהות Web Shell פעיל, ומה כן עשוי לחשוף אותו?
answer: מפני שה-Web Shell נגיש דרך HTTPS על port 443 — הפורט שהשרת חייב לקבל אליו תעבורה — והפקודות עוברות כבקשות web רגילות, לעיתים מוצפנות, שאינן ניתנות להבחנה מגלישה לגיטימית. הפרימטר לא רואה שום דבר חריג. מה שחושף אותו הוא ניטור השרת עצמו: קבצים חדשים או ששונו בספריות web, בקשות POST אל קבצים "סטטיים", ותהליך שרת שמוליד cmd.exe או ‎/bin/sh‎.
:::

## הגנה: למה פרימטר לבדו לא מספיק

מכיוון שאי אפשר לעצור את ה-Web Shell בשער, ההגנה עליו חייבת להיות רב-שכבתית —
זו דוגמה מובהקת ל-[[Defense in Depth]]. המרצה מציג שש שיטות מניעה, ואפשר לחלק אותן
לשלוש שכבות:

**למנוע את ההשתלה מלכתחילה:**

- **אימות כל ההעלאות (Validate All Uploads)** — רשימת היתר (whitelist) של סוגי
  קבצים מותרים, סריקת תוכן, אחסון ההעלאות **מחוץ ל-web root**, ושינוי שמות הקבצים
  שהועלו. זה חוסם ישירות את וקטור ה-Unrestricted File Upload.
- **פיתוח מאובטח (Secure Development Practices)** — שאילתות פרמטריות נגד SQL
  injection, אימות קלט, הימנעות מהכללות קבצים דינמיות, וביקורות קוד. זה סוגר את
  וקטור ניצול החולשות.

**להקשות ולהכיל:**

- **הקשחת השרת (Harden the Server)** — [[Least Privilege]] לתהליכי השרת, נטרול
  פונקציות מסוכנות שאינן נחוצות (‎eval‎, ‎exec‎), ושמירה על תוכנה מעודכנת ומטולאת.
  אם ה-shell כן עלה — הרשאות מינימליות מגבילות את מה שהוא יכול לעשות.
- **סגמנטציה של הרשת (Segment the Network)** — בידוד שרת ה-web מהמערכות הפנימיות
  והגבלת החיבורים היוצאים מה-[[DMZ]]. זו הבקרה שהופכת פריצת שרת בודד לאירוע מוכל
  במקום לפריצה כלל-ארגונית.

**לזהות במהירות:**

- **[[WAF]]** — חוסם חתימות מוכרות של web shells, פרמטרים חשודים ו-payloads מקודדים
  בשער האפליקטיבי, עוד לפני שהבקשה מגיעה לקובץ.
- **ניטור מתמשך (Monitor Continuously)** — כלי **ניטור שלמות קבצים
  (File Integrity Monitoring, FIM)** שמתריעים על כל קובץ חדש או ששונה בספריות
  ה-web, לצד ניתוח לוגים ו-**EDR** על השרת.

הנקודה המרכזית: זיהוי דורש ניטור מתמשך של השרת, **לא רק הגנת פרימטר**. אף שכבה
לבדה אינה אמינה — טופס העלאה אחד ששכחו לאמת מפיל את הכול. רק שילוב של מניעה,
הקשחה, סגמנטציה וזיהוי נותן הגנה אמיתית.

:::selfcheck
question: ארגון סומך על חומת אש חזקה ו-NIDS בפרימטר בלבד כהגנה מפני Web Shell. מהו הפער, ואילו שתי בקרות סוגרות אותו?
answer: הפער הוא שהפרימטר אינו רואה את ה-Web Shell כלל — הוא נגיש דרך HTTPS תקין ותעבורתו נראית רגילה. שתי בקרות משלימות סוגרות את הפער: ניטור שלמות קבצים (FIM) שמזהה את הופעת הקובץ הזדוני על השרת עצמו, וסגמנטציה של הרשת שמגבילה תעבורה יוצאת מה-DMZ ומכילה את הפגיעה גם אם ה-shell הותקן. זהו יישום של הגנה לעומק.
:::

---

:::keypoints
- Web Shell הוא סקריפט זדוני על שרת web שנותן לתוקף שליטה מרחוק דרך HTTP/HTTPS — backdoor קבוע, לא נזק חד-פעמי.
- שלושת מאפייניו: מבוסס סקריפט, מוסתר בין קבצים לגיטימיים, ושליטה מרחוק.
- הוא מגיע דרך העלאת קבצים ללא הגבלה, ניצול חולשות (RFI/LFI, SQL injection), פרטי גישה גנובים או שרשרת אספקה.
- לאחר ההשתלה: הרצת פקודות, קצירת פרטי גישה, הוצאת מידע, השחתה, התמדה — והמסוכן מכול, תנועה רוחבית מה-DMZ אל הרשת הפנימית.
- הוא הקשה ביותר לזיהוי מפני שהוא יושב במקום שבו קבצים נכתבים, נגיש דרך port 443 שאף חומת אש אינה חוסמת, ותעבורתו זהה לתעבורת web רגילה.
- לא תופסים אותו בפרימטר: ההגנה היא רב-שכבתית — אימות העלאות, פיתוח מאובטח, הקשחה, סגמנטציה, WAF וניטור שלמות קבצים.
:::
