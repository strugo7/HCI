## Question

id: q-web-shell-001
type: multiple-choice
difficulty: easy
cognitive: understand
estimatedTime: 45
points: 5
concepts:
  - Web Shell

מהי ההגדרה המדויקת ביותר ל-Web Shell?

### Answers

A. תוכנה המצפינה את קובצי השרת ודורשת תשלום כופר בעבור מפתח הפענוח.

B. נוזקה שתפקידה היחיד להוריד ולהתקין נוזקה אחרת (payload) מעבר להגנות.

C. נוזקה עצמאית המתפשטת בין שרתים ברשת ללא צורך בקובץ מארח ובלי התערבות משתמש.

D. קובץ הפעלה (‎.exe‎) שנצמד לתוכנית מארחת ורץ כשהמשתמש מפעיל אותה.

E. סקריפט זדוני המועלה לשרת web ומעניק לתוקף גישה ושליטה מרחוק דרך ממשק web סטנדרטי (HTTP/HTTPS).

Correct: E

Explanation: תשובה E היא ההגדרה של המרצה: סקריפט זדוני על שרת web הנותן שליטה מרחוק דרך HTTP/HTTPS. תשובה C מתארת **תולעת** (Worm) — עצמאית ומתפשטת, בעוד ש-Web Shell הוא פסיבי וממתין. תשובה B מתארת **Dropper** — מנגנון הפצה, לא דלת אחורית. תשובה A מתארת **Ransomware**. תשובה D מתארת **וירוס מבוסס הפעלה** — הזקוק לקובץ מארח ולהפעלת משתמש, בעוד ש-Web Shell רץ על השרת ומופעל מרחוק דרך בקשת web. הבלבול הנפוץ הוא לראות בכל נוזקה "אותו דבר"; כל אחת מוגדרת לפי המקום שבו היא פועלת ואופן ההפעלה שלה.

Learning Objective: להגדיר Web Shell ולהבחין בינו לבין תולעת, דרופר, כופרה ווירוס.

Misconception: סטודנטים ממזגים את כל סוגי הנוזקה להגדרה אחת, ואינם מבחינים בין נוזקה שרצה על תחנת הקצה לבין דלת אחורית שרצה על השרת.

## Question

id: q-web-shell-002
type: attack-analysis
difficulty: hard
cognitive: analyze
estimatedTime: 75
points: 5
concepts:
  - Web Shell

חברת אבטחת מידע מפתחת מערכת לזיהוי איומים המשלבת חתימות (Signature) וניתוח התנהגות (Behavior). איזה מהאיומים הבאים יהיה הקשה ביותר לזיהוי באמצעות שתי השיטות הללו?

### Answers

A. Web Shell המוסווה כקובץ לוג לגיטימי על שרת ה-web.

B. תוכנת כופר המצפינה קבצים ומשנה את סיומותיהם באופן המוני.

C. תולעת המנצלת פרצת אבטחה ידועה כדי להתפשט ברשת.

D. דרופר המוריד payload משרת מרוחק מיד עם הפעלתו.

E. וירוס המתפשט דרך קובצי Word עם macros.

Correct: A

Explanation: זו השאלה שנבחנה ב-2024. תשובה A נכונה כי ל-Web Shell מוסווה מצטברים שלושה יתרונות שמנטרלים גם חתימות וגם התנהגות: הוא יושב במקום שבו קבצים נכתבים ממילא (קובץ לוג אינו חריג), הוא נגיש דרך HTTPS על port 443 שאף חומת אש אינה חוסמת, ותעבורתו זהה לתעבורת web רגילה — כך שמנקודת מבט הרשת שום דבר אינו נראה חריג. תשובה E — לווירוס מאקרו יש חתימות מוכרות והתנהגות בולטת (מסמך שמריץ קוד). תשובה C — תולעת מייצרת דפוס רשת חריג וקולני שזיהוי התנהגות מצטיין בו. תשובה B — כופרה מייצרת התנהגות רועשת במיוחד: הצפנה המונית ושינוי סיומות, קלה יחסית לזיהוי. תשובה D — דרופר מבצע C2 beaconing והורדה מרחוק, דפוסים שניתוח רשת מזהה. המלכודת: להעריך "קושי לזיהוי" לפי תחכום הקוד, במקום לפי המיקום והערוץ שבהם התעבורה נראית לגיטימית.

Learning Objective: להסביר מדוע Web Shell מוסווה חומק גם מזיהוי חתימות וגם מזיהוי התנהגות.

Misconception: סטודנטים מדרגים "קושי לזיהוי" לפי מורכבות הנוזקה, ולא לפי העובדה שהתעבורה עוברת בערוץ ובמיקום שבהם היא נראית תקינה.

## Question

id: q-web-shell-003
type: scenario
difficulty: medium
cognitive: apply
estimatedTime: 70
points: 5
concepts:
  - Web Shell

בהקשר התרחיש, מהי דרך ההשתלה שהתוקף ניצל, ומהי הבקרה הישירה שהייתה מונעת אותה?

### Scenario

בית חולים מפעיל פורטל שבו מטופלים מעלים מסמכים רפואיים סרוקים. אנשי הפיתוח בדקו את
סיומת הקובץ בדפדפן בלבד. תוקף עקף את הבדיקה, העלה קובץ ‎report.php‎, וגלש אליו —
ומאותו רגע הריץ פקודות על שרת הפורטל.

### Answers

A. Compromised Credentials; אכיפת אימות רב-שלבי (MFA) על פאנל הניהול של הפורטל.

B. Unrestricted File Upload; אימות סוג הקובץ בצד השרת, אחסון העלאות מחוץ ל-web root ושינוי שם הקובץ.

C. Supply Chain Attack; בדיקת חתימות דיגיטליות של כל תוסף צד-שלישי לפני התקנתו.

D. RFI; חסימת הכללת קבצים מרוחקים בהגדרות שרת האפליקציה.

E. SQL injection; שימוש בשאילתות פרמטריות (parameterized queries) בכל פנייה לבסיס הנתונים.

Correct: B

Explanation: התרחיש מתאר במדויק **Unrestricted File Upload**: טופס העלאה שאינו מאמת את סוג הקובץ בצד השרת, ולכן מקבל קובץ ‎.php‎ הרץ. הבקרה הישירה היא אימות סוג הקובץ בשרת (whitelist), אחסון ההעלאות מחוץ ל-web root כך שלא ניתן להריצן, ושינוי שמות הקבצים. תשובה E (SQL injection) שגויה — לא הייתה כאן פנייה לבסיס נתונים אלא העלאת קובץ. תשובה A (פרטי גישה גנובים) שגויה — התוקף לא השתמש בפרטי התחברות אלא ניצל את הטופס. תשובה C (שרשרת אספקה) שגויה — הקובץ הועלה על ידי התוקף, לא הוטמע בתוסף לגיטימי. תשובה D (RFI) קרובה אך שגויה — לא הוכלל קובץ מרוחק, אלא הועלה קובץ מקומי; המיטיגציה שלה גם אינה מטפלת בטופס ההעלאה. המלכודת: בדיקת סיומת בצד הלקוח אינה בקרה — קל לעקוף אותה.

Learning Objective: לזהות בתרחיש את וקטור ההשתלה של Web Shell ולהתאים לו את בקרת המניעה הנכונה.

Misconception: סטודנטים מניחים שבדיקת סיומת בצד הלקוח מספיקה, ומבלבלים בין וקטור העלאת קבצים לבין ניצול חולשות בסיס נתונים.

## Question

id: q-web-shell-004
type: architecture
difficulty: hard

estimatedTime: 85

concepts:

  - Lateral Movement

בנק גילה Web Shell פעיל על שרת האתר הציבורי שלו, הממוקם ב-DMZ. צוות התגובה ניקה
את הקובץ. מהי המסקנה הנכונה ביותר לגבי ההיערכות ההגנתית של הבנק?

### Answers

A. יש להעביר את שרת האתר מה-DMZ אל הרשת הפנימית המוגנת יותר.

B. ניקוי הקובץ סוגר את האירוע; אם השרת נקי, האיום הוסר במלואו.

C. הבקרה הקריטית היא הגבלת התעבורה היוצאת מה-DMZ אל הרשת הפנימית, כדי למנוע תנועה רוחבית — וייתכן שהתוקף כבר התבסס בנקודות נוספות.

D. יש להוסיף חומת אש היקפית חזקה יותר, שכן היא הייתה חוסמת את ה-Web Shell מלכתחילה.

E. הפתרון הוא להצפין את כל התעבורה אל שרת האתר, כדי שהתוקף לא יוכל לתקשר עם ה-shell.

Correct: C

Explanation: תשובה C נכונה: היכולת המסוכנת ביותר של Web Shell היא תנועה רוחבית — שימוש בשרת ה-DMZ שנפרץ כקרש קפיצה אל הרשת הפנימית. לכן הגבלת התעבורה היוצאת מה-DMZ היא הבקרה שמכילה את הפגיעה, וההנחה חייבת להיות שהתוקף אולי כבר בנה התמדה (חשבונות backdoor, shells נוספים) — ניקוי קובץ בודד אינו מבטיח שהאיום הוסר. תשובה B שגויה בדיוק מסיבה זו: Key Takeaway של המרצה הוא ש"תגובה לאירוע חייבת לכלול ציד web shells". תשובה D שגויה — חומת אש היקפית אינה יכולה לחסום Web Shell, שכן הוא נגיש דרך HTTPS תקין שהשרת חייב לקבל. תשובה E שגויה — הצפנה מגנה על סודיות התעבורה, אך אינה מונעת מהתוקף לפנות ל-shell; היא אף מקשה על זיהוי. תשובה A שגויה ומסוכנת — שרת ציבורי חייב להיות ב-DMZ דווקא כדי שפריצתו לא תיתן גישה ישירה לרשת הפנימית; העברתו פנימה מבטלת את כל הרציונל של ה-DMZ.

Learning Objective: להעריך תגובה לאירוע Web Shell ולזהות שהכלה תלויה בסגמנטציה ובהגבלת תעבורה יוצאת מה-DMZ, לא בניקוי הקובץ בלבד.

Misconception: סטודנטים סוגרים אירוע ברגע שהקובץ הזדוני נמחק, ומחמיצים את התנועה הרוחבית ואת ההתמדה שהתוקף כבר ביסס.

## Question

id: q-web-shell-005
type: comparison
difficulty: medium
cognitive: analyze
estimatedTime: 65
points: 5
concepts:
  - Web Shell
  - WAF

ארגון מבקש לזהות במהירות Web Shell שהותקן על שרת ה-web שלו. איזו מהאפשרויות הבאות היא הבקרה היעילה ביותר למטרה זו?

### Answers

A. אנטי-וירוס מבוסס חתימות בלבד על תחנות הקצה של העובדים.

B. הצפנת TLS על כל התקשורת מול השרת.

C. NIDS מבוסס-רשת שבודק את כל התעבורה הנכנסת אל השרת.

D. ניטור שלמות קבצים (FIM) שמתריע על כל קובץ חדש או ששונה בספריות ה-web.

E. חומת אש היקפית עם מדיניות deny-by-default על תעבורה נכנסת.

Correct: D

Explanation: תשובה D נכונה: Web Shell הוא בראש ובראשונה **קובץ חדש או ששונה** על השרת, ולכן ניטור שלמות קבצים (FIM) הוא הבקרה הישירה שמזהה את הופעתו — זו אחת משיטות הזיהוי שהמרצה מדגיש. תשובה C (NIDS) שגויה — התעבורה אל ה-shell עוברת כ-HTTPS על port 443 ונראית כמו גלישה רגילה, כך שניתוח רשת מתקשה להבחין בה. תשובה B (הצפנה) מגנה על סודיות התעבורה אך אינה מזהה כלום, ואף מקשה על ניתוח. תשובה E (חומת אש היקפית) אינה יכולה לחסום או לזהות תעבורה אל שרת שכל תפקידו לקבל HTTPS. תשובה A (אנטי-וירוס על תחנות קצה) שגויה — ה-Web Shell יושב על השרת, לא על תחנות העובדים. הערה: [[WAF]] הוא בקרה משלימה שחוסמת חתימות של shells בשער, אך לזיהוי shell שכבר יושב על השרת, FIM הוא הכלי הישיר.

Learning Objective: לבחור את בקרת הזיהוי הנכונה ל-Web Shell ולהבין מדוע ניטור השרת עדיף על ניטור הפרימטר.

Misconception: סטודנטים מניחים שכלי רשת ופרימטר מזהים Web Shell, ואינם מבחינים שהוא נתפס דרך ניטור השרת עצמו (שינויי קבצים, תהליכים חריגים).
