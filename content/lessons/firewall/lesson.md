# חומת אש — סינון מנות וטבלאות כללים

## למה בכלל צריך חומת אש?

רשת ארגונית מחוברת לאינטרנט דרך נקודת מעבר אחת או כמה. דרך הצינור הזה זורמת כל
התעבורה הלגיטימית — גלישה, דואר, עדכונים — אבל גם כל ניסיון תקיפה מבחוץ. בלי שום
בקרה, כל מחשב באינטרנט יכול לפנות ישירות אל כל מחשב ברשת הפנימית, אל כל פורט, בכל
פרוטוקול.

[[Firewall]] היא התשובה הבסיסית ביותר לבעיה הזו: להציב שומר בנקודת המעבר, שיבדוק
כל מנה (packet) מול רשימת כללים ויחליט אם להעביר אותה או להשליך אותה. זו אבן היסוד
של הגנת ה-[[Perimeter]] והשכבה הראשונה כמעט בכל ארכיטקטורת [[Defense in Depth]].

בשיעור הזה נתמקד בחומת האש כפי שהמרצה מלמד אותה — מה שהוא מכנה שוב ושוב
**"Firewalls – layer 3"**: חומת אש שמסננת מנה-אחר-מנה לפי כותרת המנה. המוקד המרכזי
יהיה היכולת שהמרצה מתאמן עליה שלוש פעמים: **לקרוא טבלת כללים ולהחליט אם מנה נתונה
תעבור או תיחסם**.

:::objectives
- לזהות מה חומת אש בשכבה 3 בודקת בכל מנה — ומה היא לעולם אינה בודקת.
- להבחין בין סוגי הפריסה: Host-based, Network-based, Stand-alone ו-Cloud.
- לקרוא טבלת ACL ולהחליט, שורה-אחר-שורה, אם מנה נתונה מותרת או נחסמת.
- להסביר את ההבדל בין Implicit Deny (allow list) ל-Implicit Allow (block list).
- להבין מדוע חומת אש היקפית לבדה אינה מספיקה.
:::

## מה חומת אש בודקת — ומה לא

חומת אש בשכבה 3 מקבלת החלטת forward או drop על סמך ארבעה שדות בלבד, כולם בכותרת
המנה:

- **Source IP address** — מהיכן המנה הגיעה.
- **Destination IP address** — לאן היא מיועדת.
- **Port** — לאיזה שירות (80 ל-HTTP, 25 ל-SMTP, 23 ל-Telnet, 3389 ל-RDP).
- **Protocol** — TCP, UDP וכדומה.

המרצה מסכם את פעולתה במשפט שהוא צובע באדום: היא **"filters packet-by-packet"** את
התעבורה. השאלה המנחה בכל מנה היא: *"Should arriving packet be allowed in? Departing
packet let out?"*.

הנקודה הקריטית היא מה שמחוץ לרשימה: חומת אש בשכבה 3 **אינה קוראת את תוכן המנה
(payload)**. היא בודקת את המעטפה, לא את המכתב. מנה זדונית שנשלחת אל פורט מותר תיראה
לה תקינה לחלוטין. בחינת התוכן עצמו — SQL Injection, XSS וכדומה — היא תפקידו של
[[WAF]] בשכבה 7, נושא נפרד.

חומת האש שלנו היא גם **חסרת-מצב (Stateless)**: היא שופטת כל מנה בפני עצמה, בלי לזכור
מנות קודמות. חומת אש שזוכרת את מצב החיבור ([[Stateful Inspection]]) ומנועים מתקדמים
יותר שייכים למשפחת [[NGFW]] ואינם נלמדים ביחידה זו.

:::warning
טעות נפוצה: "יש לי חומת אש, אז נוזקה לא תיכנס". חומת אש בשכבה 3 אינה מזהה נוזקה —
היא לא פותחת את הקובץ. אם הנוזקה מגיעה בתעבורה אל פורט מותר (למשל הורדה דרך 443),
היא עוברת. זיהוי נוזקה הוא תפקיד של Antivirus או [[WAF]], לא של חומת אש בשכבה 3.
:::

:::selfcheck
question: מנה זדונית מגיעה מכתובת מותרת אל פורט 443 (HTTPS) של שרת פנימי, אך גוף המנה מכיל מתקפת SQL Injection. האם חומת אש בשכבה 3 תעצור אותה?
answer: לא. חומת אש בשכבה 3 בודקת רק כתובת מקור, כתובת יעד, פורט ופרוטוקול — כולם תקינים כאן. היא אינה קוראת את תוכן המנה, ולכן ה-SQL Injection חומק ממנה. עצירת מתקפה כזו דורשת בחינת תוכן בשכבה 7, כלומר WAF.
:::

## איפה מציבים אותה — סוגי פריסה

חומת אש אינה חייבת להיות קופסה אחת בכניסה לרשת. המרצה מבחין בין כמה סוגי פריסה,
ושתי ההבחנות הראשונות הן החשובות:

- **Host-based firewall** — תוכנת חומת אש המותקנת על מחשב בודד, ומגנה על **אותו
  מחשב בלבד**. חומת האש של Windows היא דוגמה.
- **Network-based firewall** — **שילוב חומרה ותוכנה** המוצב בגבול הרשת ומגן על
  **רשת שלמה** בבת אחת.
- **Stand-alone firewall** — רכיב חומרה ייעודי (appliance) המוקדש כולו לתפקיד.
- **Cloud firewall** — חומת אש המסופקת כשירות בענן, מגינה על משאבים בענן.

ההבחנה בין Host-based ל-Network-based אינה "או-או". ארגון מפעיל את שתיהן יחד: חומת
אש רשתית בפרימטר חוסמת את רוב האיומים, וחומות אש מבוססות-מארח על השרתים מוסיפות
שכבה פנימית — בדיוק רעיון ה-[[Defense in Depth]]. זו גם התשובה שהמבחן מחפש בשאלת
"איך לחלק תקציב שדרוג חומות אש": פורסים חומת אש רשתית בפרימטר **וגם** תוכנת חומת אש
על השרתים.

:::selfcheck
question: מהו ההבדל המהותי בין Host-based firewall ל-Network-based firewall, ולמה ארגון ירצה את שתיהן?
answer: Host-based היא תוכנה על מחשב יחיד ומגנה עליו בלבד; Network-based היא שילוב חומרה ותוכנה בגבול הרשת ומגנה על רשת שלמה. ארגון משלב את שתיהן כדי ליצור הגנה רב-שכבתית: אם תוקף חדר מעבר לפרימטר, חומת האש שעל השרת עדיין חוסמת אותו.
:::

## ACL — טבלת הכללים, שורה אחר שורה

הלב של חומת אש בשכבה 3 הוא ה-**Access Control List (ACL)** — רשימת כללים מסודרת
שכל שורה בה קובעת האם סוג מסוים של מנה מותר (ALLOW) או חסום (DENY). לכל שורה יש
ערכים לשדות שחומת האש בודקת. כך נראית טבלת ה-ACL שהמרצה מציג:

| Permission | IP Address | Protocol | Destination | Port |
|---|---|---|---|---|
| ALLOW | 162.213.214.14 | TCP | 10.10.10.2 | 80 |
| ALLOW | 54.21.66.112 | TCP | ANY | 80 |
| DENY | 192.168.1.1 | TCP | ANY | 80 |
| ALLOW | 65.252.1.2 | TCP | ANY | 80 |
| DENY | ANY | TCP | ANY | 80 |
| ALLOW | ANY | TCP | ANY | 80 |
| DENY | ANY | UDP | 10.10.10.1 | 23 |
| DENY | 255.255.255.0 | TCP | ANY | 25 |
| ALLOW | 10.10.0.1 | TCP | ANY | 110 |

הערך **ANY** הוא כרטיס כללי — הוא מתאים לכל ערך בשדה. כדי לקרוא את הטבלה עוברים על
המנה הנבדקת מול כל שורה ומחפשים התאמה: כתובת המקור, הפרוטוקול, היעד והפורט. השורה
הראשונה שמתאימה קובעת את הגורל — ALLOW או DENY.

המרצה מפרק את המיומנות לשתי מיומנויות קטנות. הראשונה — **חסימה לפי כתובת IP**:

| Permission | IP Address | Protocol | Destination | Port |
|---|---|---|---|---|
| ALLOW | 162.213.214.140 | TCP | ANY | 80 |
| ALLOW | 54.21.66.112 | TCP | ANY | 80 |
| DENY | 40.55.130.66 | TCP | ANY | 80 |

כאן שלוש כתובות מגיעות אל חומת האש. 162.213.214.140 ו-54.21.66.112 מותרות; לעומתן
40.55.130.66 נופלת על שורת ה-DENY ונחסמת — למרות שהיא פונה לאותו פורט 80 כמו
האחרות. הכתובת היא שקבעה, לא הפורט.

השנייה — **חסימה לפי פורט**:

| Permission | IP Address | Protocol | Destination | Port |
|---|---|---|---|---|
| ALLOW | ANY | TCP | ANY | 80 |
| ALLOW | ANY | TCP | ANY | 25 |
| ALLOW | ANY | TCP | ANY | 110 |
| DENY | ANY | UDP | ANY | 23 |
| DENY | ANY | TCP | ANY | 3389 |

כאן כל הכתובות שוות (ANY), והפורט הוא שמכריע. פורטים 80, 25 ו-110 מותרים מכל
כתובת; פורט 23 (Telnet) ופורט 3389 (RDP) חסומים מכל כתובת. מנה אל פורט 3389 תיחסם
גם אם הגיעה ממקור לגיטימי לחלוטין.

:::important
סדר השורות מכריע. חומת אש עוצרת בשורה הראשונה שמתאימה למנה ומיישמת אותה — היא אינה
ממשיכה לחפש. כלל ALLOW רחב שממוקם לפני כלל DENY ספציפי ינטרל אותו. לכן כותבים תמיד
את הכללים הספציפיים לפני הכללים הכלליים.
:::

:::selfcheck
question: לפי טבלת "חסימה לפי פורט", מנה TCP מכתובת 8.8.8.8 אל פורט 3389 — מותרת או נחסמת?
answer: נחסמת. השורה "DENY · ANY · TCP · ANY · 3389" מתאימה למנה (פרוטוקול TCP, פורט 3389, כל כתובת מקור). מקור המנה אינו רלוונטי כאן — הפורט 3389 חסום מכל מקור.
:::

## Implicit Deny מול Implicit Allow

מה קורה למנה שלא התאימה לאף שורה בטבלה? כאן נכנסת אחת ההחלטות החשובות ביותר בתכנון
חומת אש — **מדיניות ברירת המחדל**. המרצה מציג שני מודלים הפוכים:

| מודל | ברירת המחדל | סוג הרשימה | הכללים המפורשים |
|---|---|---|---|
| **Implicit Deny** | מה שלא הותר במפורש — **נחסם** | whitelist / allow list | רשימת ALLOW |
| **Implicit Allow** | מה שלא נחסם במפורש — **מותר** | blacklist / block list | רשימת DENY |

**Implicit Deny** ("deny by default") הוא ההפוך של אמון: הרשימה מכילה רק אישורים
מפורשים, וכל השאר חסום. זו רשימת היתר — **allow list** (whitelist). זו גישת ברירת
המחדל הבטוחה, כי כל מה שלא חשבתם עליו — חסום.

**Implicit Allow** ("allow by default") הוא ההפך: הרשימה מכילה רק חסימות מפורשות,
וכל השאר עובר. זו רשימת חסימה — **block list** (blacklist). היא נוחה אך מסוכנת: כל
איום שלא הוספתם אליה במפורש — עובר בחופשיות.

:::analogy
Implicit Deny הוא מסיבה עם רשימת אורחים — מי שלא ברשימה לא נכנס. Implicit Allow הוא
מסיבה פתוחה עם רשימת מורחקים — כל אחד נכנס, חוץ ממי שנמצא ברשימה השחורה. במסיבה עם
רשימת אורחים, אורח לא-צפוי נחסם אוטומטית; במסיבה הפתוחה, פורץ שלא הכרתם מראש נכנס.
:::

:::tip
בבחינת טבלת ACL, חפשו קודם אם קיימת שורת "catch-all" בסוף (למשל DENY · ANY · ANY).
שורה כזו הופכת את הטבלה למפורשת ומגדירה את מדיניות ברירת המחדל שלה בשחור-על-גבי-לבן.
:::

:::selfcheck
question: ארגון בנה טבלת ACL עם ALLOW בלבד, ובסופה שורת "DENY · ANY · ANY · ANY · ANY". איזה מודל זה, ומה קורה למנה אל פורט חדש שלא נחשב?
answer: זהו Implicit Deny (allow list / whitelist). מנה אל פורט שלא הותר במפורש נופלת על שורת ה-DENY הכללית בסוף ונחסמת. זו התנהגות ברירת המחדל הבטוחה — מה שלא הותר במפורש, נחסם.
:::

## למה חומת אש לבדה אינה מספיקה

חומת אש היקפית עוצרת את רוב התעבורה הזדונית — אבל לא את כולה. המרצה ממחיש זאת
בתרחיש **Firewall Breaches**: תוקף מגביר את עוצמת התקיפה, וחלק מהנוזקות מצליחות
לעבור את חומת האש הרשתית ולחדור פנימה. חומות האש מבוססות-המארח שעל התחנות עדיין
חוסמות חלק מהתקיפות, אך אחת התחנות בכל זאת נדבקת.

הלקח הוא בדיוק הרעיון של [[Defense in Depth]]: אף שכבה אינה מושלמת, ולכן בונים כמה
שכבות. חומת אש בשכבה 3 מסננת לפי כתובות ופורטים; [[WAF]] מוסיף בחינת תוכן בשכבה 7;
[[DMZ]] מבודד את השירותים הציבוריים מהרשת הפנימית; ומערכות זיהוי מתריעות על מה
שחמק. חומת האש היא ההתחלה של ההגנה, לא סופה.

:::selfcheck
question: אם חומת אש היקפית כבר חוסמת את רוב התקיפות, מדוע להתקין גם חומות אש מבוססות-מארח על השרתים הפנימיים?
answer: מפני שחומת האש ההיקפית אינה מושלמת — חלק מהתקיפות חומקות דרכה, וברגע שתוקף בפנים אין דבר שיעצור אותו מלנוע בין השרתים. חומות אש מבוססות-מארח מוסיפות שכבת הגנה פנימית שממשיכה לחסום גם אחרי שהפרימטר נפרץ. זהו עקרון ה-Defense in Depth.
:::

---

:::keypoints
- חומת אש בשכבה 3 (Packet Filtering) בודקת ארבעה שדות בכותרת: Source IP, Destination IP, Port, Protocol — ולעולם לא את תוכן המנה.
- סוגי פריסה: Host-based (מחשב בודד), Network-based (רשת שלמה), Stand-alone (חומרה ייעודית), Cloud.
- ACL היא רשימת כללים מסודרת; חומת האש עוצרת בשורה הראשונה שמתאימה. הסדר מכריע.
- Implicit Deny = allow list (whitelist): מה שלא הותר, נחסם — ברירת המחדל הבטוחה.
- Implicit Allow = block list (blacklist): מה שלא נחסם, מותר — נוח אך מסוכן.
- חומת אש היקפית לבדה אינה מספיקה — היא שכבה אחת ב-Defense in Depth.
:::
