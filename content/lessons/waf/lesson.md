# WAF — שכבה 7: כשבודקים את תוכן החבילה, לא רק את הכותרת

## חבילה אחת, שני קוראים

תוקף שולח לאתר הבנק את המחרוזת `' OR '1'='1` דרך שדה שם המשתמש. זו התקפת SQL
Injection — פקודת מסד נתונים זדונית שמתחזה לקלט תמים. עכשיו שאלה: כיצד היא הגיעה
לשרת?

התשובה מטרידה. היא הגיעה על פני **פורט 443**, בדיוק כמו כל גולש לגיטימי באתר.
מבחינת [[Firewall]] קלאסי — זה שהמרצה ממקם במפורש ב**שכבה 3** — החבילה הזו מושלמת:
כתובת מקור חוקית, יעד חוקי, פורט HTTPS מותר. ולכן היא **עוברת**. חומת האש לא נכשלה
ולא נפרצה. היא פשוט עשתה בדיוק את מה שהיא יודעת לעשות: קראה את הכותרת, אישרה, והמשיכה
הלאה. את **תוכן** החבילה היא מעולם לא פתחה.

זו הבעיה שכל היחידה הזו באה לפתור. יש סוג שלם של התקפות שנכנסות דרך הדלת הראשית,
בתוך תעבורה תקינה לחלוטין, ושום בדיקת כותרת בעולם לא תעצור אותן. כדי לעצור אותן
צריך רכיב ששואל שאלה אחרת לגמרי — לא "מאיפה זה בא?" אלא "**מה כתוב בפנים?**". הרכיב
הזה הוא [[WAF]].

:::objectives
- להסביר מדוע בדיקת כותרות (שכבה 3) אינה מספיקה, ומדוע מטען זדוני עובר דרך פורט מותר.
- לזהות את בדיקת התוכן בשכבה 7 בשמותיה השונים: WAF, Layer 7 firewall, Application Gateway (Proxy FW), ו-Deep Packet Inspection (DPI).
- למנות את ארבע ההתקפות שה-WAF בודק: SQL Injection, XSS, Cookie Tampering, Parameter Tampering.
- לשחזר את טבלת ההשוואה שכבה 3 מול שכבה 7 — מה כל שכבה רואה, מה היא חוסמת, ומה לא.
- להסביר את מחיר בדיקת התוכן: throughput נמוך, ותלות בסיום (termination) של TLS.
:::

## למה בדיקת הכותרת לא מספיקה

[[Firewall]] בשכבה 3 מקבל החלטה על סמך ארבעה שדות בלבד: כתובת IP מקור, כתובת IP
יעד, פורט ופרוטוקול. זה מהיר, זה זול, וזה עובד מצוין נגד סוג אחד של איום — מישהו
שמנסה להתחבר לאן שאסור לו, או מכתובת שחסומה.

אבל שים לב מה נשאר מחוץ לתמונה: **מה שכתוב בתוך החבילה**. ברגע שפורט 443 פתוח (וחייבים
לפתוח אותו — אחרת אין אתר), כל בקשה שמגיעה אליו נראית זהה מבחוץ. גולש שקורא כתבה,
לקוח שמעביר כסף, ותוקף ששולח SQL Injection — שלושתם חבילות HTTPS על פורט 443. הכותרת
שלהם זהה. ההבדל היחיד ביניהם הוא ב**תוכן**, וזה בדיוק מה שחומת האש בשכבה 3 אינה קוראת.

זו אינה מגבלה שאפשר "לכוונן". היא מובנית בשכבה שבה חומת האש פועלת. כדי לזהות SQL
Injection צריך להבין מהי בקשת SQL; כדי לזהות סקריפט זדוני צריך להבין מהו HTML. אלה
מושגים של **שכבת האפליקציה** — שכבה 7 — ורכיב שיושב בשכבה 3 פשוט לא מדבר בשפה הזו.

:::important
מטען זדוני שמגיע על פני פורט מותר הוא **בלתי נראה** לחומת אש קלאסית — לא מפני שהיא
חלשה, אלא מפני שהיא בודקת את הכותרת ולא את התוכן. זו הסיבה היחידה והמלאה לקיומו של
WAF.
:::

:::selfcheck
question: ארגון פתח בחומת האש בשכבה 3 את פורט 443 ("רק HTTPS מותר") והכריז שהאתר מוגן. תוקף שולח SQL Injection דרך טופס באתר. מה יקרה, ולמה?
answer: החבילה תעבור. מבחינת חומת האש בשכבה 3 היא תעבורת HTTPS תקינה: הכתובת חוקית והפורט (443) מותר. חומת האש קוראת רק את הכותרת, וה-SQL Injection מסתתר ב**תוכן** — בגוף הבקשה האפליקטיבית — שאותו היא כלל אינה בוחנת. רק רכיב בשכבה 7 שבודק את תוכן החבילה יכול לזהות ולעצור את ההתקפה הזו.
:::

## מה שכבה 7 רואה: Deep Packet Inspection

הרכיב שפותר את הבעיה יושב ב**שכבה 7**, שכבת האפליקציה, ובודק את מה שחומת האש בשכבה 3
דילגה עליו: את **תוכן** החבילה. המנגנון שבו הוא עושה זאת נקרא **Deep Packet
Inspection (DPI)** — בדיקה עמוקה של מטען החבילה, ולא רק של הכותרת. זה המונח שהמבחן
נוקב בו בשמו (מבחן 2025, שאלה 20), ולכן שווה לקבע אותו: **DPI = בדיקת התוכן, לא
הכותרת.**

למרצה יש כמה שמות לרעיון הזה, והוא מפריד ביניהם — אבל כולם בדיוק אותו דבר: בדיקת
תוכן בשכבה 7.

- **Layer 7 firewall** — השם הכללי. "Examines contents within data packets".
- **Application Gateway (Proxy FW)** — הצורה המוקדמת: "gateway from one network to
  another for a specific application", שסינן לפי **Traffic contents** (תוכן התעבורה)
  עבור פרוטוקולים כמו HTTP, FTP ו-SMTP.
- **WAF (Web Application Firewall)** — הגרסה הממוקדת בתעבורת אתרים (HTTP/HTTPS).

המרצה מציג את ה-WAF כמסננת תעבורת HTTP בין מקורות התעבורה לבין שרת היעד, ומונה
במפורש **ארבע** התקפות שה-Application Firewall בודק:

- **SQL Injection** — הזרקת פקודת מסד נתונים דרך שדה קלט.
- **Cross-site Scripting (XSS)** — הזרקת סקריפט זדוני שירוץ בדפדפן של קורבן אחר.
- **Cookie Tampering** — שינוי עוגייה כדי להתחזות או להסלים הרשאות.
- **Parameter Tampering** — שינוי פרמטר בבקשה (למשל מחיר או מזהה משתמש) כדי לעקוף
  לוגיקה עסקית.

ארבעתן חולקות תכונה אחת: כולן מגיעות בתוך תעבורה שנראית לגיטימית לחלוטין בשכבה 3.
אף אחת מהן אינה נחסמת על ידי כלל פורט או כתובת. רק בדיקת התוכן — DPI — חושפת אותן.

:::selfcheck
question: מדוע "Layer 7 firewall", "Application Gateway (Proxy FW)" ו-"WAF" נחשבים אצלנו לאותו מושג, ומהו המנגנון המשותף לשלושתם?
answer: כי שלושתם עושים את אותו דבר עקרוני — בודקים את **תוכן** החבילה בשכבת האפליקציה (שכבה 7), במקום להסתפק בכותרות. המנגנון המשותף הוא Deep Packet Inspection (DPI): בדיקה עמוקה של מטען החבילה. ההבדלים ביניהם הם של דור וניסוח (Proxy FW הוא הצורה המוקדמת, WAF הוא הגרסה הממוקדת ב-HTTP), לא של עיקרון.
:::

## הטבלה: שכבה 3 מול שכבה 7

זו הטבלה שהמרצה בונה בשקופית 30, וזהו הלב של כל היחידה. ההבחנה היחידה והאמיתית שהוא
עושה בכל הדק אינה בין "דורות" של חומות אש — אלא בין **שכבה 3 (כותרות) לשכבה 7
(תוכן)**. שווה לדעת אותה שורה־שורה:

| | Layer 3 firewalls | Layer 7 firewalls |
| --- | --- | --- |
| **Traffic filtering** | Based on IP addresses and ports | Examines contents within data packets |
| **Performance** | High throughput; handles large volumes of traffic | Lower throughput due to deep packet inspection |
| **Speed of inspection** | Fast; simple filtering based on addresses & ports | Slower; detailed analysis of packet contents |
| **Security focus** | General threats; basic access control | Complex, application-targeted threats |
| **Use case** | Suitable where speed and volume are critical | Needed for sensitive applications needing deep inspection |
| **Typical threat mitigation** | Blocks threats based on source/destination info | Mitigates threats exploiting application vulnerabilities like SQL injections, XSS |

אם נזקק את הטבלה לארבע השאלות שבאמת נשאלות עליה — **מה כל שכבה רואה, מה היא חוסמת,
מה היא לא, והיכן היא יושבת** — מקבלים את התמונה הזו:

| | שכבה 3 — [[Firewall]] | שכבה 7 — [[WAF]] |
| --- | --- | --- |
| **מה רואה** | כותרות: IP מקור/יעד, פורט, פרוטוקול | תוכן החבילה: הבקשה האפליקטיבית עצמה |
| **מה חוסם** | תעבורה מכתובת/פורט אסורים; איומים כלליים | SQL Injection, XSS, Cookie/Parameter Tampering |
| **מה לא רואה / לא חוסם** | מטען זדוני בתוך פורט מותר | נפחים גדולים במחיר סביר; תעבורה מוצפנת עד לפענוח |
| **היכן יושב** | גבול הרשת, בשכבת הניתוב | לפני שרתי ה-Web, בין ה-CDN ל-Load Balancer |

בשקופית שלפני הטבלה (29) המרצה מצייר את שתי השכבות זו לצד זו בזרימה אחת: תעבורה
יוצאת מהסניף עוברת קודם דרך **Layer 7 Firewall** המסומן "App filter" (בודק תוכן),
ואחריו דרך **Layer 3 router** המסומן "IP filter" (בודק כתובות), ורק אז לאינטרנט.
שתי השכבות בשרשרת אחת — לא במקום זו את זו.

:::warning
אל תבלבל בין "שכבה 7" ל"טובה יותר". שכבה 7 אינה שדרוג של שכבה 3 — היא בודקת **דבר
אחר**. שכבה 3 מהירה וזולה ומטפלת בנפחים; שכבה 7 יסודית ויקרה ומטפלת בתוכן. ארגון
אמיתי צריך את **שתיהן**, וזו בדיוק הסיבה שקיים [[NGFW]] — הרכיב שמאחד את שתי השכבות
בקופסה אחת.
:::

:::selfcheck
question: לפי טבלת שכבה 3 מול שכבה 7, מהו ההבדל ב"Traffic filtering" בין שתי השכבות, ואיזו שכבה תזהה SQL Injection?
answer: שכבה 3 מסננת "Based on IP addresses and ports" — כלומר לפי כותרות בלבד. שכבה 7 "Examines contents within data packets" — בודקת את תוכן החבילה. SQL Injection מגיע בתוך תוכן הבקשה על פני פורט מותר, ולכן רק שכבה 7 (WAF) תזהה אותו; שכבה 3 תעביר אותו כי מבחינת הכותרות הוא תקין.
:::

## המחיר: בדיקת תוכן איטית, יקרה, ומוצפנת

אם שכבה 7 רואה כל כך הרבה יותר, למה לא לשים אותה בכל מקום ולוותר על שכבה 3? כי לבדיקת
התוכן יש מחיר אמיתי, והמרצה מציין אותו במפורש בטבלה.

**היא איטית ויקרה.** לפתוח כל חבילה ולנתח את מטענה דורש הרבה יותר עבודה מלקרוא כותרת.
לכן ל-Layer 7 firewall יש **throughput נמוך יותר** — "Lower throughput due to deep
packet inspection" — והוא מתאים למקומות שבהם צריך בדיקה עמוקה, לא למקומות שבהם צריך
לספוג נפחי תעבורה עצומים במהירות. שם עדיין שולטת שכבה 3.

**היא חייבת לסיים את ה-TLS.** זו הנקודה המעשית שהכי קל לפספס. תעבורת אתרים מודרנית
מוצפנת (HTTPS). ל-WAF, חבילה מוצפנת נראית כרעש חסר משמעות — הוא לא יכול לבדוק תוכן
שהוא לא יכול לקרוא. לכן WAF **חייב לסיים (terminate) את ה-TLS**: לפענח את התעבורה
אצלו, לבדוק את התוכן הגלוי, ורק אז להעביר אותה הלאה. בלי סיום ה-TLS, כל ה-DPI הופך
חסר תועלת — יש לו על מה להסתכל רק אחרי הפענוח.

:::warning
מכאן נובעת מלכודת מבחן קלאסית (בסגנון שאלה 20, 2025): כשהפרוטוקול המותר עצמו מנוצל
להברחת נוזקה, הפתרון **אינו** לבטל את ההצפנה כדי "לראות טוב יותר" — ביטול הצפנה פוגע
ישירות בסודיות ובשלמות. הפתרון הוא **DPI וסינון אפליקטיבי**: החומה מסיימת את ה-TLS
בצורה מבוקרת, בודקת את התוכן, וממשיכה. כל מסיח שמציע "לבטל הצפנה" או "להתעלם מהתוכן
כדי לא לפגוע בביצועים" הוא שגוי.
:::

:::selfcheck
question: מדוע WAF חייב "לסיים TLS" (TLS termination) כדי לבצע את תפקידו, ומה קורה אם לא?
answer: כי בדיקת תוכן (DPI) דורשת לקרוא את התוכן, ותעבורת HTTPS מוצפנת. כל עוד החבילה מוצפנת, ה-WAF רואה רעש חסר משמעות ולא יכול לזהות SQL Injection או XSS. לכן הוא חייב לפענח (לסיים) את ה-TLS אצלו, לבדוק את התוכן הגלוי, ורק אז להעביר הלאה. בלי סיום ה-TLS ה-WAF עיוור — יש לו יכולת בדיקה, אבל אין לו על מה להפעיל אותה.
:::

## היכן זה מתחבר: WAF כשכבת הגנה

WAF אינו עומד לבדו. הוא אחת השכבות בעיקרון ה-[[Defense in Depth]]: שכבה 3 חוסמת מה
ששכבה 7 בזבזנית מכדי לבדוק, שכבה 7 חוסמת מה ששכבה 3 עיוורת אליו, וכל שכבה נוספת
מצמצמת את מה שעבר את הקודמת. [[NGFW]] הוא הביטוי המרוכז של הרעיון: הוא מאחד סינון
מנות בשכבה 3 עם בדיקת תוכן בשכבה 7 (המרצה מונה WAF במפורש בין יכולותיו) בקופסה אחת.

הדוגמה המובהקת לערך של WAF היא [[Web Shell]] — סקריפט זדוני שתוקף מעלה לשרת דרך
חולשת אפליקציה, ומשתמש בו כדלת אחורית. ההעלאה עצמה מגיעה כבקשת HTTP תקינה על פורט
מותר; חומת אש בשכבה 3 לא תראה בה דבר חריג. WAF, שבודק את תוכן הבקשה, הוא קו ההגנה
המרכזי נגד ניסיון כזה — הוא יכול לזהות את דפוס ההעלאה הזדוני או את הפקודות הנשלחות
ל-Web Shell קיים, בדיוק מפני שהוא קורא את מה ששכבה 3 מדלגת עליו.

:::selfcheck
question: מדוע WAF נחשב לקו הגנה מרכזי נגד Web Shell, בעוד חומת אש בשכבה 3 חסרת ערך מולו?
answer: כי Web Shell מועלה ומופעל דרך תעבורת HTTP/HTTPS תקינה על פורט מותר. חומת אש בשכבה 3 בודקת כותרות בלבד ולכן רואה בקשה לגיטימית לחלוטין. WAF בודק את **תוכן** הבקשה (DPI), ולכן יכול לזהות את דפוס ההעלאה הזדוני או את הפקודות הנשלחות ל-Web Shell — מה שנמצא בדיוק בשכבה שחומת האש הקלאסית עיוורת אליה.
:::

---

:::keypoints
- מטען זדוני על פורט מותר (SQL Injection על 443) **בלתי נראה** לחומת אש בשכבה 3 — היא בודקת כותרת, לא תוכן.
- **WAF = Layer 7 firewall = Application Gateway (Proxy FW)** — שלושה שמות, רעיון אחד: בדיקת **תוכן** החבילה בשכבה 7.
- המנגנון נקרא **Deep Packet Inspection (DPI)** — בדיקה עמוקה של המטען. זהו המונח שהמבחן נוקב בו בשמו.
- ה-WAF בודק ארבע התקפות: **SQL Injection, Cross-site Scripting (XSS), Cookie Tampering, Parameter Tampering**.
- טבלת שכבה 3 מול 7: שכבה 3 = כתובות/פורטים, throughput גבוה, מהיר, איומים כלליים. שכבה 7 = תוכן, throughput נמוך, איטי, איומים אפליקטיביים.
- המחיר: בדיקת תוכן **איטית ויקרה**, ו-WAF **חייב לסיים TLS** כדי לראות תעבורה מוצפנת.
- שכבה 7 אינה מחליפה את שכבה 3 — היא משלימה אותה. [[NGFW]] מאחד את שתיהן; [[Defense in Depth]] הוא העיקרון.
:::
