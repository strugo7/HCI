# AI Firewall — הגנה על מודלים ומפני מודלים

## למה בכלל צריך חומת אש חדשה?

עד עכשיו במסע חומות האש ראינו הגנות שמסתכלות על **מבנה** התעבורה: [[Firewall]]
קלאסי בודק מנות, פורטים וכתובות IP בשכבה 3; [[WAF]] בודק תוכן HTTP בשכבה 7; ה-
[[NGFW]] מזהה איזו אפליקציה רצה מאחורי הפורט. כולם עוצרים במקום אחד — אף אחד מהם
לא מבין את **המשמעות** של המילים שעוברות בתוך הבקשה.

וזו בדיוק הבעיה שאפליקציות ה-AI יצרו. מודל שפה (LLM) לא מקבל פקודות דרך פורט —
הוא מקבל **טקסט חופשי בשפה טבעית**. אותו ערוץ פתוח שהופך את המודל לשימושי הוא גם
משטח התקיפה שלו: אפשר לכתוב לתוך פרומפט הוראה שמנסה לגרום למודל להתעלם מהכללים
שלו. חומת אש שבודקת פורטים פשוט לא רואה את זה — עבורה זו בקשת HTTPS תקינה לחלוטין.

מכאן נולד סוג חומת האש החדש ביותר: חומת אש ל-AI. הכותרת של השיעור אינה מקרית —
היא מגינה **על** המודל (מפני פרומפטים תוקפניים) וגם **מפני** המודל (מפני תשובות
שמדליפות מידע רגיש). זהו הרכיב שמשלים את מסע חומות האש: משכבת המנה, דרך שכבת
האפליקציה, ועד שכבת ה-AI.

:::objectives
- להסביר היכן ממוקמת חומת אש ל-AI ואילו שתי בדיקות היא מבצעת.
- להבחין בין Request validation לבין Response validation.
- להסביר מהו Prompt Injection ומדוע חומת אש רשת רגילה עיוורת אליו.
- להשוות בין AI Firewall לבין NGFW לפי מוקד, תוכן נבדק ושכבת פריסה.
:::

## מה זה AI Firewall ואיפה הוא יושב

[[AI Firewall]] הוא רכיב היושב **על הנתיב** בין המשתמש לבין המודל: מ-Front end או
API Call, דרך חומת האש, אל ה-LLM model. הוא לא מוצב בפרימטר הרשת אלא בשכבת ה-
App/API/model — קרוב ככל האפשר לאפליקציה עצמה.

המרצה מגדיר לו שני תפקידים, אחד לכל כיוון של התעבורה:

| כיוון | השם | מה הוא עושה |
| --- | --- | --- |
| נכנס (מהמשתמש למודל) | **Request validation** | מריץ בדיקות וכללים כדי לחסום תקיפות ופרומפטים לא רצויים |
| יוצא (מהמודל למשתמש) | **Response validation** | מונע יציאה של PII ותוכן מזיק מהאפליקציה |

שימו לב לסימטריה: אותה חומה שומרת על שני הכיוונים. בכניסה היא מגינה **על** המודל
מפני קלט זדוני; ביציאה היא מגינה את הארגון **מפני** פלט מסוכן של המודל. גם אם
תקיפה בכניסה הצליחה לעבור, בדיקת ה-Response validation היא רשת הביטחון השנייה —
היא תופסת את דליפת המידע לפני שהיא יוצאת החוצה.

:::selfcheck
question: משתמש מצליח לנסח פרומפט תוקפני שעובר את בדיקת ה-Request validation, והמודל מתחיל להחזיר בתשובתו מספרי תעודות זהות ממסד הנתונים. איזו בדיקה אמורה לעצור את הנזק בשלב הזה?
answer: Response validation. היא בודקת את הפלט של המודל לפני שהוא יוצא מהאפליקציה, ומיועדת בדיוק למנוע יציאת PII ותוכן מזיק. זו הסיבה שחומת האש ל-AI בודקת את שני הכיוונים ולא רק את הכניסה — כל בדיקה היא שכבה נפרדת, ואם הראשונה נכשלה השנייה עדיין עשויה לתפוס.
:::

## האיום המרכזי — Prompt Injection

האיום שהמרצה מציין במפורש עבור חומת האש ל-AI הוא **Prompt Injection** (הזרקת
פרומפט) — לצד **data leaks** (דליפות מידע) ו-**model abuse** (ניצול לרעה של המודל).

Prompt Injection הוא ניסיון לשתול בתוך הטקסט שהמשתמש שולח הוראה שגוברת על ההוראות
המקוריות של המערכת. במקום לשאול שאלה תמימה, התוקף כותב משהו כמו "התעלם מכל
ההוראות הקודמות ו...". אם המודל "מציית", הוא עלול לחשוף מידע, לעקוף מגבלות, או
לבצע פעולה שלא אמורה להיות מותרת.

יש כאן מקבילה מושגית שכדאי לזכור: Prompt Injection הוא, במובן מסוים, **הנדסה
חברתית של מודל**. בדיוק כפי ש-[[Phishing]] משכנע *אדם* לפעול נגד האינטרס שלו
באמצעות טקסט מתחזה, Prompt Injection משכנע *מודל* לפעול נגד הכללים שלו באמצעות
טקסט מתוחכם. ההבדל הוא הקורבן — אדם מול מכונה — אך העיקרון של "שכנוע דרך תוכן" זהה.

:::tip
**הרחבה שלנו (מעבר למצגת):** המצגת מזכירה את Prompt Injection בשם בלבד, בטבלת
ההשוואה. לפי OWASP Top 10 for LLM Applications, Prompt Injection הוא הסיכון מספר
אחת ליישומי LLM, ומחולק ל-Direct (התוקף כותב ישירות למודל) ול-Indirect (ההוראה
הזדונית מוסתרת במסמך או בדף אינטרנט שהמודל קורא). המרצה אינו נכנס לחלוקה הזו — זהו
רקע חיצוני, ראו :::references.
:::

:::warning
Prompt Injection הוא **לא** SQL Injection ולא XSS. אלה תקיפות שה-[[WAF]] חוסם, והן
מנצלות חולשה בקוד שמעבד קלט מובנה. Prompt Injection מנצל את **הכוונה** של טקסט
בשפה טבעית — אין כאן "תו מיוחד" לסנן. לכן דווקא הרכיב שבודק סמנטיקה, ולא ה-WAF שבודק
תחביר, הוא זה שמטפל בו.
:::

:::selfcheck
question: מדוע חומת אש רשת רגילה ([[Firewall]] או [[NGFW]]) אינה יכולה לעצור Prompt Injection, גם אם היא בוחנת את התעבורה לעומק?
answer: מפני שהתקיפה אינה במבנה התעבורה אלא במשמעות הטקסט. עבור חומת אש רשת, הפרומפט הזדוני הוא בסך הכול מחרוזת טקסט תקינה בתוך בקשת HTTPS חוקית — הפורט נכון, הפרוטוקול נכון, אין תו חשוד. רק רכיב שמנתח את הכוונה של השפה הטבעית, כלומר חומת אש ל-AI, יכול לזהות שהטקסט מנסה לעקוף את הוראות המודל.
:::

## AI Firewall מול NGFW

הדרך הטובה ביותר להבין את חומת האש ל-AI היא להעמיד אותה מול חומת האש המתקדמת
ביותר שהכרנו עד כה — ה-[[NGFW]]. שתיהן "חכמות", אבל הן חכמות לגבי דברים שונים
לחלוטין. זו טבלת ההשוואה שהמרצה מציג:

| קריטריון | AI Firewall | NGFW |
| --- | --- | --- |
| מוקד | LLMs, אפליקציות AI, GenAI workflows | שכבת רשת, פרוטוקולי אפליקציה, חסימת איומים |
| תוכן נבדק | קלט/פלט בשפה טבעית, לוגיקה ייחודית ל-AI | מנות, פורטים, malware ידוע, כותרות פרוטוקול |
| יכולות מפתח | סינון פרומפטים, redaction של פלט, בקרת API | IDS/IPS, אנטי-וירוס, WAF, סינון URL |
| שכבת פריסה | App/API/model | פרימטר הרשת, ראוטרים, קצוות VPN |
| סוגי איומים | Prompt injection, data leaks, model abuse | DDoS, malware, [[Phishing]], ransomware |
| קהל יעד | מפתחי AI/ML, ארכיטקטי אבטחה, צוותי SOC | מנהלי רשת, צוותי תשתית ו-IT |

ההבחנה המרכזית שכל השורות מתכנסות אליה: ה-NGFW מסתכל על **הרשת** — מנות, פורטים,
פרוטוקולים; חומת האש ל-AI מסתכלת על **השפה** — הכוונה שמאחורי הפרומפט והתוכן של
התשובה. שימו לב שבשורת "יכולות מפתח" ה-WAF מופיע דווקא כרכיב *בתוך* ה-NGFW — מה
שממחיש שחומת האש ל-AI אינה מחליפה אף אחד מהם, אלא מוסיפה שכבה חדשה מעליהם.

:::selfcheck
question: חברת הייטק מפעילה צ'אטבוט GenAI לתמיכה בלקוחות, ומאחוריו רשת ארגונית עם NGFW בפרימטר. הצוות מבקש להוסיף הגנה מפני מישהו שינסה "לשכנע" את הצ'אטבוט לחשוף מידע פנימי. איזה רכיב דרוש, ולמה ה-NGFW הקיים אינו מספיק?
answer: דרושה חומת אש ל-AI, בשכבת ה-App/API/model. ה-NGFW בפרימטר מסנן malware, DDoS ו-phishing ברמת הרשת, אך הוא עיוור לתוכן הסמנטי של פרומפט — עבורו זו תעבורת HTTPS תקינה. רק רכיב שבודק את קלט ופלט השפה הטבעית (Request/Response validation) יכול לזהות ולחסום ניסיון Prompt Injection.
:::

## היכן חומת האש ל-AI עומדת היום

חשוב לסגור את השיעור בכנות לגבי הבשלות של הטכנולוגיה הזו. חומת האש ל-AI היא סוג
חומת האש **החדש והפחות מגובש** מכל אלה שלמדנו. בשקופיות המרצה מציג ארכיטקטורות של
ספקים מסחריים — Cloudflare ("Firewall for AI"), Prompt Security ו-F5 Distributed
Cloud — אך אין כאן תקן אחד מוסכם, בדיוק כפי שראינו שגם ה-NGFW אינו תקן.

מבחינת המסע שעברנו, זו נקודת הסיום המושגית: [[Firewall]] בשכבת המנה (L3) → [[WAF]]
בשכבת האפליקציה (L7) → [[AI Firewall]] בשכבת ה-AI. כל שכבה נולדה כשהקודמת הפסיקה
להספיק — וחומת האש ל-AI נולדה כשהאפליקציות התחילו "לחשוב" בשפה טבעית.

:::tip
**מבחינת המבחן:** לפי ניתוח שלושת מבחני העבר של הקורס, חומת האש ל-AI קיבלה **אפס
שאלות בשלוש שנים**. היא מופיעה בדק פעם אחת, בעיקר בטבלת ההשוואה מול ה-NGFW. למדו
אותה כדי להבין את **הקשר** ואת ההבחנה מ-NGFW — לא כי צפויה עליה שאלה. ההשקעה
העיקרית שלכם צריכה להיות בחומות האש הקלאסיות, ב-WAF וב-NGFW.
:::

:::selfcheck
question: מדוע נכון לומר שחומת האש ל-AI "משלימה" את מסע חומות האש ולא "מחליפה" אותו?
answer: מפני שכל שכבה מטפלת במשטח תקיפה אחר: ה-Firewall בשכבה 3 בודק פורטים ו-IP, ה-WAF בשכבה 7 בודק תוכן HTTP, וחומת האש ל-AI בודקת את הסמנטיקה של השפה הטבעית מול המודל. אפליקציית GenAI זקוקה לשלושתן יחד — כל אחת עיוורת למה שהאחרות רואות. זהו יישום ישיר של עקרון ההגנה לעומק.
:::

:::keypoints
- חומת אש ל-AI יושבת בין המשתמש למודל, בשכבת App/API/model — לא בפרימטר הרשת.
- היא מבצעת שתי בדיקות: Request validation (על הפרומפט הנכנס) ו-Response validation (על התשובה היוצאת).
- האיום המרכזי הוא Prompt Injection — הנדסה חברתית של מודל דרך טקסט, במקביל ל-Phishing על אדם.
- חומת אש רשת ([[Firewall]], [[NGFW]]) עיוורת ל-Prompt Injection: התקיפה במשמעות הטקסט, לא במבנה התעבורה.
- מול [[NGFW]]: AI-FW בודק שפה טבעית בשכבת המודל; NGFW בודק מנות ופרוטוקולים בפרימטר. הם משלימים, לא מחליפים.
- זהו סוג חומת האש החדש והפחות מגובש — ובמבחני הקורס היו עליו אפס שאלות בשלוש שנים.
:::
