# WAF

:::definition
WAF (Web Application Firewall) הוא רכיב הגנה הבוחן את **תוכן** החבילה בשכבת
האפליקציה (שכבה 7), ולא רק את כותרות הרשת (IP ופורט) כמו [[Firewall]] קלאסי.
המנגנון שבו הוא עושה זאת נקרא **Deep Packet Inspection (DPI)** — בדיקה עמוקה של
מטען החבילה — וכך הוא מזהה וחוסם התקפות אפליקטיביות כגון **SQL Injection** ו-**Cross-site
Scripting (XSS)** גם כשהן מגיעות בתוך תעבורת HTTP/HTTPS תקינה לכל דבר.
:::

:::warning
למונח WAF **אין מונח עברי מקובל** — המרצה משתמש במונח האנגלי בלבד. במבחן הוא יופיע
כ-WAF, כ-Layer 7 firewall, כ-Application Gateway (Proxy FW) או בשם המנגנון שלו,
Deep Packet Inspection (DPI). ארבעת אלה הם שמות שהמרצה מפריד ביניהם, אך כולם
מבטאים רעיון אחד: **בדיקת תוכן החבילה בשכבה 7**.
:::

## הסבר פשוט

חומת אש רגילה קוראת רק את ה"מעטפה" של החבילה: מאיזו כתובת היא באה, לאן היא הולכת,
דרך איזה פורט. היא לא פותחת את המעטפה. כל עוד הכתובת והפורט מותרים — החבילה עוברת,
ולא משנה מה כתוב בתוכה.

WAF פותח את המעטפה. הוא בוחן את **התוכן** — את הבקשה האפליקטיבית עצמה — ושואל האם
בתוך תעבורת האתר התקינה מסתתרת התקפה. תוקף ששולח פקודת מסד נתונים זדונית (SQL
Injection) דרך טופס באתר עושה זאת על פני פורט 443, בדיוק כמו כל גולש לגיטימי; חומת
אש קלאסית תעביר אותה בלי היסוס, כי מבחינת הכותרות היא נראית תקינה. WAF הוא הרכיב
היחיד שרואה את התוכן ולכן היחיד שיכול לעצור אותה.

## הסבר טכני

WAF יושב בשכבת האפליקציה של מודל השכבות (Layer 7), בניגוד ל-[[Firewall]] שהמרצה
ממקם במפורש בשכבה 3. ההבדל אינו "חזק יותר" אלא **מה נבדק**: שכבה 3 בודקת כתובות
ופורטים (headers); שכבה 7 בודקת את מטען החבילה (contents). המרצה מציג את WAF
כמסננת תעבורת HTTP בין מקורות התעבורה לבין שרת היעד, וממקם אותה ארכיטקטונית בין
ה-CDN לבין ה-Load Balancer — לפני שרתי ה-Web ומסד הנתונים.

המרצה מונה ארבעה סוגי התקפות שה-Application Firewall בודק: **SQL Injection**,
**Cross-site Scripting (XSS)**, **Cookie Tampering** ו-**Parameter Tampering**.
הצורה המוקדמת של הרעיון היא ה-**Application Gateway (Proxy FW)** — "gateway from
one network to another for a specific application" — שסינן לפי **תוכן התעבורה**
(Traffic contents) עבור פרוטוקולים כמו HTTP, FTP ו-SMTP.

לבדיקת התוכן יש מחיר, וזו הנקודה שהמרצה מדגיש: ה-DPI **איטי ויקר** מבחינת משאבים
(throughput נמוך יותר), ו-WAF **חייב לסיים (terminate) את ה-TLS** כדי לראות משהו —
תעבורה מוצפנת נראית לו כרעש חסר משמעות עד שהיא מפוענחת.

:::example
אתר בנק חושף טופס התחברות על פני HTTPS (פורט 443). תוקף מזין בשדה שם המשתמש את
המחרוזת `' OR '1'='1` — התקפת SQL Injection. מבחינת [[Firewall]] בשכבה 3 זו חבילת
HTTPS תקינה לחלוטין: כתובת מקור חוקית, יעד חוקי, פורט 443 מותר — ולכן היא עוברת.
ה-WAF, לעומת זאת, מפענח את ה-TLS, בוחן את תוכן הבקשה, מזהה את דפוס ה-SQL Injection
בשדה, וחוסם את הבקשה לפני שהיא מגיעה למסד הנתונים.
:::

:::warning
WAF אינו מחליף את חומת האש הקלאסית — הוא משלים אותה בשכבה אחרת. שכבה 3 עוצרת מה
ששכבה 7 בזבזנית מכדי לבדוק (נפחי תעבורה גדולים, בקרת גישה בסיסית), ושכבה 7 עוצרת
מה ששכבה 3 עיוורת אליו (תוכן זדוני בתוך פורט מותר). [[NGFW]] הוא הרכיב שמאחד את
שתי השכבות בקופסה אחת.
:::

:::diagram
תרשים זרימה אופקי של תעבורת HTTP נכנסת: מקורות התעבורה (משתמשי אינטרנט) → CDN →
תיבת **WAF (Layer 7)** → Load Balancer → שרתי Web → Database. מעל תיבת ה-WAF כיתוב
"Deep Packet Inspection — בודק תוכן, לא רק כותרת". ארבעה חצים נכנסים אל ה-WAF
ומסומנים באדום ונחסמים: SQL Injection, Cross-site Scripting (XSS), Cookie
Tampering, Parameter Tampering. חץ חמישי, בקשת HTTP לגיטימית, עובר בירוק אל
ה-Load Balancer. בפינה הערה: "כדי לבדוק תוכן — ה-WAF חייב לסיים TLS".
:::
