# Proxy Firewall

:::definition
Proxy Firewall (המכונה גם Application-Level Gateway או ALG) הוא חומת אש הפועלת
בשכבת האפליקציה (שכבה 7). במקום להעביר מנות בין שני צדדים, היא **מנתקת (terminates)
את חיבור הלקוח ופותחת חיבור שני, נפרד, אל השרת בשם הלקוח**. הלקוח והשרת לעולם אינם
מתקשרים ישירות — כל תעבורה עוברת דרך ה-Proxy, שמרכיב מחדש את הבקשה האפליקטיבית ובוחן
את מלוא תוכנה לפני שהוא מעביר אותה הלאה.
:::

## הסבר פשוט

חומת אש קלאסית ([[Firewall]]) היא כמו שומר שמעביר מכתבים מיד ליד: הוא בודק את
הכתובת על המעטפה ומחליט אם להעביר, אבל המכתב עצמו ממשיך מהשולח אל הנמען. Proxy
Firewall עובד אחרת לגמרי. הוא **עוצר את המכתב**, פותח אותו, קורא את כולו, ורק אם
הכול תקין הוא כותב מכתב **חדש** משלו ושולח אותו הלאה. השולח המקורי והנמען לעולם לא
נגעו באותו מכתב.

התוצאה היא ניתוק מוחלט של החיבור הישיר. אין ולו מנה אחת שעוברת מהלקוח אל השרת כמו
שהיא — תמיד יש באמצע מתווך שקרא, הבין, והרכיב מחדש. זה בדיוק מה שמאפשר ל-Proxy
לראות את מה שחומת אש רגילה עיוורת אליו: לא רק את הכתובת, אלא את **התוכן**.

## הסבר טכני

> הרחבה שלנו: המרצה מזכיר את Application Gateway (Proxy FW) בשקופית אחת בלבד (דק 05,
> שקופית 25) כ-"early type of firewall device that acts as a gateway from one
> network to another for a specific application". ההסבר המנגנוני שלהלן נשען על
> מקורות חיצוניים (CompTIA Security+ SY0-701, NIST SP 800-41 Rev. 1, ותיעוד יצרנים)
> ומחובר לעוגן היחיד של המרצה — הבחנת שכבה 3 מול שכבה 7 (שקופית 30).

הרעיון המרכזי הוא **ניתוק החיבור (connection termination)**. כאשר לקוח פונה לשרת דרך
Proxy, נוצרים שני חיבורי TCP נפרדים: הראשון בין הלקוח ל-Proxy, והשני בין ה-Proxy
לשרת. ה-Proxy מסיים את הראשון, מפרק את הבקשה עד לשכבה 7, מבין את הפרוטוקול (HTTP,
FTP, SMTP וכדומה), בודק אותה מול מדיניות, ורק אז יוזם בעצמו את החיבור השני. מכיוון
שה-Proxy מדבר את הפרוטוקול, הוא יכול לאכוף **התאמה לפרוטוקול (protocol conformance)**
ולחסום בקשות מעוותות שחומת אש מבוססת-כותרות כלל לא הייתה מבחינה בהן.

Proxy מגיע בשני כיוונים: **Forward Proxy** יושב מול הלקוחות ומתווך את פנייתם החוצה
(שליטה ביציאה, סינון אתרים, אכיפת מדיניות גלישה), ו-**Reverse Proxy** יושב מול השרת
ומתווך את הפניות הנכנסות אליו מבחוץ — ומסתיר את השרת האמיתי מאחוריו. ה-Reverse Proxy
מוצב לרוב ב-[[DMZ]], כך שהתוקף מדבר עם המתווך ולא נוגע בשרת הפנימי.

המחיר של הגישה הזו כפול. ראשית, הבדיקה יקרה: פירוק והרכבה מחדש של כל בקשה מוסיפים
**השהיה (latency)** ומורידים את קצב התעבורה. שנית, כדי לראות תוכן מוצפן ה-Proxy
**חייב לסיים את ה-TLS** (TLS termination) — לפענח, לבדוק, ולהצפין מחדש; ללא זה התוכן
נראה לו כרעש. שלישית, ה-Proxy הוא **מודע-לפרוטוקול (per-protocol)**: הוא זקוק למנוע
ייעודי לכל פרוטוקול שהוא מבין, ולכן אינו מכסה פרוטוקול שלא נכתב עבורו במפורש.

:::example
ארגון מציב Forward Proxy ליציאה לאינטרנט. עובד מנסה להוריד קובץ דרך HTTP; הבקשה
נעצרת ב-Proxy, שמסיים את החיבור, בוחן את בקשת ה-HTTP המלאה, מזהה שהיעד ברשימת
החסימה, ומסרב — מבלי שנפתח אי־פעם חיבור בין תחנת העובד לשרת החיצוני. השרת מעולם לא
ראה את כתובת ה-IP של העובד; הוא ראה רק את ה-Proxy.
:::

:::warning
Proxy Firewall אינו זהה ל-[[WAF]], אף שקל לבלבל. ה-Proxy הוא ה**שער הכללי בשכבת
האפליקציה** — המתווך שמנתק את החיבור עבור כל פרוטוקול. ה-WAF הוא **צאצא מתמחה** של
הרעיון, שממוקד בתעבורת web (HTTP/HTTPS) ובחתימות התקפה אפליקטיביות כמו SQL Injection.
שניהם בוחנים תוכן בשכבה 7; ה-Proxy הוא הקטגוריה הרחבה, וה-WAF הוא ההתמחות ל-web.
:::
