# CONTENT_SPEC.md

Version: 1.0

---

# Purpose

This document defines how educational content is structured inside the platform.

It is the single source of truth for every lesson, concept, quiz and learning object.

All educational content generated by Claude MUST follow this specification.

The frontend assumes every document follows these rules.

---

# Educational Philosophy

This platform does not teach through slides.

It teaches through understanding.

The objective is not memorization.

The objective is building a connected mental model.

Every lesson should help students answer:

- What is it?
- Why is it important?
- How does it work?
- Where is it used?
- What should I remember?

---

```
Course

↓

Lesson

↓

↓

↓

Learning Components
```

---

# Course

A course contains lessons.

Example

```
Computer Security
```

---

# Lesson

A lesson teaches one major subject.

Examples

```
Cyberspace

Malware

Firewall

Identity Management
```

Each lesson contains

- Metadata
- Introduction
- Objectives
- Sections
- Summary
- References

---

# Section

A lesson consists of multiple sections.

A section teaches ONE concept.

Good

```
What is Cyberspace?
```

Bad

```
Cyberspace, Internet, Cloud and Firewalls
```

Never teach multiple unrelated concepts inside one section.

---

# Topic

A topic is a subsection.

Example

```
Cyberspace

↓

Physical Layer

↓

Hardware Components
```

---

# Concepts

Every important concept exists exactly once.

Examples

```
Firewall

Router

DNS

Cloud

Availability

Integrity

Confidentiality

Payload
```

Concepts live inside

```
content/concepts
```

Lessons reference concepts.

Lessons never redefine concepts.

---

# Concept Requirements

Every concept must contain

Definition

Simple Explanation

Technical Explanation

Example

Related Concepts

Diagram Suggestion

Appears In Lessons

---

# Definitions

Definition blocks answer

"What is this?"

Definitions should

- be short
- precise
- objective

Definitions never contain examples.

---

# Explanations

Explanation blocks answer

"How does it work?"

Explanations may include

- context
- history
- technical details

---

# Examples

Every important concept requires at least one example.

Examples should

- come from real life
- use familiar systems
- avoid unnecessary complexity

---

# Diagrams

Every abstract concept should contain a diagram suggestion.

Never embed images.

Instead describe

- what should be drawn
- labels
- flow
- important elements

---

# Images

Images should support learning.

Images should never be decorative.

---

# Videos

Videos are optional.

Describe educational videos.

Do not embed YouTube URLs.

---

# Analogies

Analogies are optional.

Use them only when they simplify difficult concepts.

---

# Common Mistakes

Every difficult topic should include common misunderstandings.

Example

Firewall ≠ Antivirus

Internet ≠ Cyberspace

Authentication ≠ Authorization

---

# Important Notes

Use Important blocks only for facts students must remember.

Avoid overusing them.

---

# Self Check

Every major section must end with a Self Check.

Questions should verify understanding.

Not memorization.

---

# Quiz

Quizzes are independent objects.

Lessons only reference quizzes.

Quiz files live inside

```
content/quizzes
```

---

# Flashcards

Flashcards are independent objects.

Lessons only reference them.

---

# Writing Style

Always write in Hebrew.

Keep English names for technologies.

Examples

Firewall

DNS

TCP/IP

Cloud

OAuth

Never translate technology names unnecessarily.

---

# Writing Principles

Always

Explain first.

Example second.

Summary last.

Never assume previous knowledge unless listed as prerequisite.

---

# Paragraph Length

Prefer short paragraphs.

Maximum

4-6 lines.

Avoid large text blocks.

---

# Lists

Prefer bullet lists whenever possible.

---

# Tables

Use tables only for comparisons.

---

# Visual Learning

Whenever a visual explanation would improve understanding,

include

```
:::
...
:::
```

---

# Progressive Learning

Lessons should build knowledge gradually.

Simple

↓

Intermediate

↓

Advanced

Never jump directly to advanced explanations.

---

# Concept References

Always reference concepts using Obsidian links.

Example

```
[[Firewall]]

[[DNS]]

[[Cloud]]
```

Never use Markdown links.

---

# Knowledge Graph

Concept references automatically build the knowledge graph.

Do not manually describe graph relationships inside lessons.

---

# Lesson Completion Checklist

Every lesson must contain

✅ Introduction

✅ Learning Objectives

✅ Definitions

✅ Explanations

✅ Examples

✅ Diagram Suggestions

✅ Self Check

✅ Summary

✅ Related Concepts

If any of these are missing,

the lesson is incomplete.

---

# Reusability

Never duplicate educational content.

Definitions belong to Concepts.

Lessons reference Concepts.

Quizzes reference Concepts.

Flashcards reference Concepts.

Everything should be reusable.

---

# Separation of Responsibilities

Content

↓

Parser

↓

Knowledge Objects

↓

Renderer

↓

React Components

Content never controls presentation.

Presentation never modifies educational content.

---

# Future Compatibility

All content written using this specification must remain compatible with future versions of the platform.

The specification should evolve without requiring existing lessons to be rewritten.

Backward compatibility should always be preserved.
